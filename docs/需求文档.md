# 📘 Offer-Catcher 项目全景需求与架构复盘文档
**文档属性**：系统设计复盘 / 需求基线 (Baseline)
**项目定位**：基于混合 RAG 与 Multi-Agent 架构的“面经收集与自适应对练”智能体。
**核心受众**：项目开发者（用于开发导航与面试复盘）。

---

## 💡 一、 需求背景与痛点溯源 (Background & Pain Points)

### 1.1 业务背景
作为一名准备“Agent/大模型应用开发”岗位的求职者，在日常复习中重度依赖小红书、牛客网等社交平台的面经分享。但当前获取和消化面经的效率极低。

### 1.2 核心痛点 (User Pain Points)
1. **输入端（脏数据多）**：面经多为图片长截图，夹杂大量无用社交信息，非结构化，极难搜索。
2. **处理端（时效性差，幻觉重）**：AI 技术迭代极快（如最新出的 MCP 协议），直接用预训练大模型回答面经题极易产生幻觉或输出过时内容；且面经中混杂大量“针对个人简历”的深挖题，大模型强答会产生“脏数据”。
3. **输出端（缺乏全局视角与闭环）**：缺乏对“目标公司高频考点”的宏观统计；缺乏真实的模拟面试交互与“掌握度”追踪闭环。

---

## 🎯 二、 核心需求定义与解决方案 (Core Requirements & Solutions)

为了解决上述痛点，系统需求被定义为四大核心模块，并分别设计了针对性的技术方案：

### 2.1 知识无感录入与智能清洗 (Ingestion & Cleansing)
* **需求**：用户直接发送截图，系统自动提取结构化数据，并过滤不合理的生成请求。
* **解决方案**：
  * **多模态提取引擎**：调用 Vision LLM（如 GPT-4o）将图片解析为标准 JSON。
  * **分类与熔断机制（亮点）**：在提取时强制对题目打标。分为 `knowledge`（客观题）、`project`（项目深挖题）、`behavioral`（行为题）。对于后两者触发**熔断**，系统只存题目，绝不瞎编答案，节约 Token 并保证知识库纯净度。
  * **Payload 设计**：存储时添加 `created_at` 时间戳字段，便于后续数据分析和按时间排序。
  * **静态词表对齐**：使用本地字典将”鹅厂”强转为”腾讯”，保证后续图谱统计的数据一致性。

### 2.2 异步标准答案生成 (Async Answer Generation)
* **需求**：批量解析数十道面经题时，系统需自动去网上搜索最新资料并生成标准答案，且不能阻塞用户前端界面的响应。
* **解决方案**：
  * **消息队列削峰（RabbitMQ）**：将需要回答的客观题打包（含公司、岗位等完整上下文）发送至 RabbitMQ。防止并发请求触发 LLM 厂商的 API 限流（HTTP 429）。
  * **Multi-Agent 架构**：剥离出专门的 `Web_Search_Agent` 作为后台 Worker 消费队列，执行“联网搜索 -> 归纳总结 -> 写入数据库”的闭环。

### 2.3 混合检索与宏观统计 (Hybrid RAG)
* **需求**：既能精准搜索某一道具体的题目，又能宏观统计出“字节最近常考的 3 个技术栈”。
* **解决方案**：
  * **Qdrant 向量数据库（精搜）**：采用“Question-Level”单题细粒度切块，并在文本前进行 **上下文拼接 (Context Enrichment)**。利用 Qdrant 的 Payload 建立标量索引，实现高并发下的预过滤混合检索。
  * **知识图谱与夜间批处理（统计）**：采用图数据库（如 Neo4j）双写考点实体。放弃入库时的实时智能体消歧（太慢），改用 **Nightly Batch（凌晨定时任务）** 调用大模型进行离线图谱节点合并，保证体验与质量的平衡。

### 2.4 主动对练与状态机流转 (Proactive Push & State Machine)
* **需求**：系统需像教练一样，主动推送未掌握的题目，并在用户作答后进行判分和进度更新。
* **解决方案**：
  * **三档熟练度状态机**：设计 `mastery_level` 为 0（完全不会）、1（比较熟悉）、2（已掌握）三档。系统依据此字段做 Top-K 推题排序。
  * **打分智能体 (Scoring Agent)**：
    * **触发拦截**：利用 IM 客户端的原生“引用回复”功能作为强上下文标记，直接唤醒打分 Agent。
    * **基准比对**：从 Qdrant 拉取异步生成的“标准答案”作为基准，对比用户的口语化回答，生成反馈意见并执行状态跃迁（Upsert 数据库）。

## ⚖️ 三、 架构推演与核心权衡复盘 (Architecture Trade-offs)
*本章节是你面试时最核心的谈资，记录了“为什么这么做，而不是那么做”。*

### 3.1 为什么选 Qdrant 而不是 PGVector？
* **复盘**：PGVector 适合重度依赖关系型数据的传统业务。本项目核心在于**“基于元数据的高频过滤（如筛选公司、熟练度）”**。Qdrant 底层由 Rust 编写，原生支持 Payload 高效标量索引，且提供极度平滑的本地开发体验（基于文件/内存的存储模式），更契合 Agentic Router 频繁的结构化条件查询需求。

### 3.2 为什么引入 RabbitMQ？直接协程异步不行吗？
* **复盘**：初期设想是直接开 Python `asyncio`。但在极端场景下（用户连发 5 张包含百道题目的截图），协程会导致瞬间并发极高，极易触发下游 LLM 的 `Rate Limit` 导致任务静默失败丢失。引入 RabbitMQ 实现了真正的**系统解耦与削峰填谷**，并通过死信队列（DLX）保障了长流程异步任务的可靠性。

### 3.3 为什么用“引用回复”而不是“LLM 自动意图判断”来处理答题？
* **复盘**：如果所有对话都让 LLM 去猜意图，不仅 Token 消耗巨大，且在用户多轮穿插对话（既答题又搜新知识）时极易发生“上下文混淆”。**妥协与智慧**：利用通讯软件的天然 UI 交互（引用机制）作为**硬状态机触发器**，构建旁路网关，只有在非引用的“自由对话”下才交给 LLM Router 处理。实现了极高的稳定性和零误判率。

## 🗺️ 四、 MVP 边界与演进路线图 (Evolution Roadmap)

为确保项目如期落地，需求被严格划分为四个演进阶段：

* **🔥 Phase 0 & 1：核心数据流 (已完成) ✅**
  * **核心产出**：
    * Vision Extractor Agent（多模态提取 + 分类熔断）
    * Qdrant 向量数据库（支持 Upsert、混合检索、Payload 索引）
    * RabbitMQ 消息队列（异步削峰填谷）
    * Answer Specialist Agent（联网搜索生成答案）
    * React 前端（Next.js 16 + Ant Design）

* **🚀 Phase 1.5：性能优化与异步架构 (已完成) ✅**
  * **核心产出**：
    * 两阶段检索架构（向量召回 + Rerank 精排）
    * Redis 缓存层（TTL + 主动失效 + 延迟双删）
    * 异步面经解析系统（任务队列 + Worker）
    * 题目聚类功能（KMeans + 聚类筛选）
    * 服务端过滤（避免内存溢出）

* **🧠 Phase 2：高级 RAG 与业务闭环 (规划中)**
  * **核心产出**：引入图谱和 Nightly Batch 脚本；实现 0-1-2 打分状态机与 `Scoring Agent` 的对比判分逻辑。

* **🌐 Phase 3：外部接入与体验优化 (规划中)**
  * **核心产出**：接入 OpenClaw / 微信体系；配置早晚定时器 (Cron)；引入 Redis 语义缓存提升系统响应速度。

## 📝 五、 项目复盘总结 (PM's Summary)

“Offer-Catcher”早已超越了一个简单的”调用大模型 API 的脚本”，它成长为了一个具备 **消息驱动、多智能体协同、混合检索引擎以及离线批处理能力** 的标准现代企业级后端架构。

在整个需求推演过程中，我们坚持了三个极其宝贵的工程原则：
1. **拥抱数据现实**：面对特异性强烈的项目类问题，果断采用“分类熔断”，体现了对大模型局限性的深刻认知。
2. **架构读写分离**：入库极速双写，答题异步剥离，保证了系统的响应吞吐量（Throughput）。
3. **克制的 MVP 边界**：砍掉暂不紧急的“删除逻辑”，延后 Redis 缓存，将刀刃用在解决最核心的数据飞轮流转上。

带着这份深刻的复盘文档，你已经吃透了这个项目的每一根毛细血管。在未来的面试中，你可以自信地应对任何针对系统设计、并发处理和 Agent 架构的深度拷问。
