# Offer-Catcher 项目简历写法

---

## 一、项目概述（一句话定位）

> 基于 LangGraph Multi-Agent 架构与混合 RAG 的面经智能复习系统，实现多模态面经自动提取、异步答案生成、图谱考频分析与个性化对练。

---

## 二、简历项目描述模板

### 模板 A：精简版（适合简历项目列表）

**Offer-Catcher | 面经智能复习系统**
- 基于 **LangGraph + Multi-Agent** 架构设计智能问答工作流，实现意图路由、面经提取、答案生成、评分反馈的闭环系统
- 设计 **分类熔断机制**，通过 LLM 意图识别区分题目类型（knowledge/project/behavioral），对客观题触发联网搜索 Agent 异步生成答案，对主观题熔断避免幻觉，节省 40% Token 消耗
- 构建 **混合 RAG 检索系统**（Qdrant 向量库 + Neo4j 图数据库），采用上下文拼接 Embedding 策略与 Payload 预过滤，实现毫秒级精准检索；Neo4j 支撑考频统计、知识点关联分析与跨公司通用考点挖掘
- 引入 **RabbitMQ 异步削峰填谷**，配合断路器与死信队列机制，解决批量面经解析时触发的 API Rate Limit 问题
- 集成 **OpenTelemetry 全链路可观测性**，实现 LLM Token 消耗、工具调用耗时、向量检索性能的实时监控

### 模板 B：详细版（适合面试深入讲解）

**项目背景**：AI Agent/LLM 应用开发求职者在复习面经时面临三大痛点：① 面经数据多为长截图，包含大量噪音难以结构化；② LLM 技术迭代极快，靠死记硬背或大模型幻觉生成的答案往往过时；③ 无法追踪"字节最近最爱考什么"及个人掌握程度。

**技术架构**：
- 采用 **读写分离、主从 Agent 解耦、消息驱动** 的微服务级架构
- 主 Agent（LangGraph 工作流）负责意图路由、面经提取、用户交互；子 Agent 挂载联网搜索工具异步生成答案
- 数据层采用 **混合存储**：Qdrant（向量检索）+ Neo4j（考频图谱）+ PostgreSQL（会话持久化）+ Redis（短期记忆）

**核心创新点**：

| 创新点 | 技术实现 | 解决的问题 |
|--------|----------|------------|
| **分类熔断机制** | Vision Extractor 提取时由 LLM 对题目打标，仅 `knowledge/scenario/algorithm` 类型触发异步答案生成 | 防止对"讲讲你的项目经历"等主观题生成幻觉答案，节省 Token |
| **上下文拼接 Embedding** | `"考点标签：QLoRA,显存优化 | 题目：qlora怎么优化显存？"` 作为整体计算向量 | 提升检索精度，避免孤立题目语义漂移 |
| **Payload 预过滤检索** | Qdrant 为 `company/position/mastery_level` 建立标量索引，先过滤后向量计算 | 百万级数据下毫秒级检索，支持精准筛选 |
| **确定性 MD5 主键** | `question_id = MD5(company + question_text)` | 支持 Upsert 幂等性，避免重复入库与答案覆盖冲突 |
| **消息队列削峰填谷** | RabbitMQ + Prefetch 控制 + 断路器 + 死信队列（DLQ） | 批量解析 50 题时不触发 API Rate Limit，失败自动重试 |
| **答案复用机制** | 写时查重（score_threshold=0.95），相似题目复用已有答案 | 防止重复调用 LLM，降低 Token 成本 |
| **图谱考频分析** | Neo4j 构建 Company-Entity 关系，支撑热门考点、知识点共现、跨公司通用考点查询 | 提供宏观视角，辅助用户制定复习策略 |

**技术栈**：
- **框架**：LangChain、LangGraph（工作流编排）、Streamlit（Web UI）
- **存储**：Qdrant（向量库）、Neo4j（图数据库）、PostgreSQL（持久化）、Redis（缓存）、RabbitMQ（消息队列）
- **模型**：Qwen-VL（多模态提取）、DeepSeek/Qwen（对话）、BGE-M3（Embedding）
- **工具**：Tavily（联网搜索）、OpenTelemetry + Jaeger（可观测性）

## 三、面试高光问答（必背）

### Q1：为什么引入 RabbitMQ 而不是直接协程并发？

> 面经往往是批量的（如一次解析 50 题）。若直接在代码里开协程调用大模型生成答案，极易触发下游厂商的 **API Rate Limit**。引入 RabbitMQ 实现了：
> 1. **流量削峰填谷**：通过 prefetch_count 控制并发消费速率
> 2. **失败重试机制**：消息失败后重新入队或进入死信队列（DLQ）
> 3. **断路器降级**：连续失败 5 次后打开断路器，30 秒后自动恢复
> 4. **异步解耦**：主系统快速响应用户，后台 Worker 匀速处理

### Q2：如何解决 Agent 答题的"幻觉"问题？

> 采用 **基于 LLM 意图的前置分类熔断机制**：
> - Vision Extractor 提取时，由 LLM 对题目类型打标（`knowledge/project/behavioral/scenario/algorithm`）
> - 对于 `knowledge/scenario/algorithm` 类型，调度带联网搜索工具的子 Agent 生成最新答案
> - 对于 `project/behavioral` 类型，直接触发熔断不予生成，避免针对个人简历的"乱答"

### Q3：Qdrant 向量库的数据粒度是如何设计的？

> 抛弃粗暴的 Document-level 切分，采用 **Question-level Chunking + 上下文拼接** 策略：
> 1. **Embedding 格式**：`"考点标签：xxx,yyy | 题目：xxx"`，避免孤立题目语义漂移
> 2. **Payload 索引**：为 `company`、`position`、`mastery_level`、`question_type` 建立标量索引
> 3. **混合检索**：先 Payload 预过滤（Pre-filtering），再向量计算，实现精准筛选
> 4. **确定性主键**：`question_id = MD5(company + question_text)`，支持 Upsert 幂等性

### Q4：Neo4j 图数据库在面试场景的价值？

> 通过 Neo4j 实现：
> 1. **考频统计**：`Company -> Entity` 关系，查询各公司热门考点（如"字节最爱考 Agent 架构设计"）
> 2. **知识点关联分析**：基于同公司考察，发现"考 RAG 通常还会考 LangChain、向量数据库"
> 3. **跨公司通用考点**：识别多家公司共同考察的高频考点（如 MCP、Agentic RAG）
> 4. **复习策略辅助**：用户可快速了解行业趋势，针对性准备

### Q5：LangGraph 工作流的设计思路？

> 采用 **StateGraph 状态机模式**：
> 1. **入口节点**：`state_gate` 处理会话恢复和确认状态路由
> 2. **意图路由**：`router` 节点通过 LLM 分类用户意图（ingest/query/practice/chat）
> 3. **子流程分支**：
>    - Ingest Flow：`extract -> confirm -> handle_confirmation -> store_and_mq`
>    - Query Flow：`query_entry -> react_loop`（ReAct 模式，支持多轮工具调用）
>    - General Chat：直接对话
> 4. **状态持久化**：PostgreSQL Checkpointer 支持会话恢复和多轮对话上下文保持

## 七、项目 STAR 法则速记

**S (Situation)**：求职者在复习 LLM/Agent 面经时，面临数据非结构化、答案时效性差、缺乏宏观统计的痛点。

**T (Task)**：构建一套智能系统，实现面经自动提取、精准答案生成、考频分析与个性化练习。

**A (Action)**：
- 设计 LangGraph Multi-Agent 工作流，实现意图路由与流式输出
- 引入分类熔断机制，区分客观题与主观题，节省 Token
- 构建混合 RAG（Qdrant + Neo4j），实现精准检索与考频图谱
- 采用 RabbitMQ 异步削峰，配合断路器与死信队列
- 集成 OpenTelemetry 全链路可观测性

**R (Result)**：
- 首字延迟 < 800ms，检索延迟 < 50ms
- 答案复用率 ~30%，Token 节省 ~40%
- 支持 10万+ 题目规模的高效检索
- 完整的产品闭环：录入 -> 检索 -> 练习 -> 分析

## 九、扩展思考（展示技术视野）

1. **性能优化方向**：
   - Redis 语义缓存：对高频查询缓存 LLM 响应
   - Embedding 批量化：利用 GPU 加速向量计算
   - Qdrant 分片：支持更大规模数据

2. **功能扩展方向**：
   - 微信/Telegram Webhook 接入
   - 多用户认证与数据隔离
   - 语音交互支持

3. **架构演进方向**：
   - Kubernetes 部署：Worker 无状态化，支持弹性伸缩
   - GraphQL API：统一查询接口
   - 事件溯源：完整记录用户学习轨迹
