## 一、项目概述

### 1.1 项目定位

Offer-Catcher 是一个面向「AI 大模型与 Agent 应用开发」岗位的面试准备系统，核心功能包括：

- **面经解析**：从文本/图片中提取结构化面试题目
- **智能检索**：基于向量数据库的语义化题目检索
- **对话练习**：多轮对话式面试模拟
- **知识图谱**：题目聚类与知识点关联
- **长期记忆**：用户画像与学习进度管理

## 二、核心亮点与技术深度

### 2.1 LangGraph 多意图工作流引擎

#### 为什么这样设计？

传统 ChatBot 采用单一路径处理所有请求，无法区分「导入面经」、「查询题目」、「闲聊」等不同意图。本项目需要支持多种业务流程：

1. **Ingest Flow**：用户导入面经 → 提取 → 确认 → 入库
2. **ReAct Flow**：用户提问 → 工具调用 → 生成回答

#### 怎么设计的？

采用 **状态机 + 条件路由** 架构：

```
START → state_gate → router → [ingest_flow / react_flow]
                    ↓
              ingest_flow: extract → confirm → handle_confirmation → store_and_mq → END
                    ↓
              react_flow: react_loop → END
```

**核心代码结构**：

```python
# backend/app/agents/graph/workflow.py
workflow = StateGraph(AgentState)

# 节点定义
workflow.add_node("state_gate", nodes.state_gate_node)
workflow.add_node("router", nodes.router_node)
workflow.add_node("extract", nodes.extract_node)
workflow.add_node("react_loop", nodes.react_loop_node)

# 条件边
workflow.add_conditional_edges("router", edges.route_by_intent, {
    "ingest_flow": "extract",
    "react_flow": "react_loop",
})
```

**状态设计**：

```python
class AgentState(TypedDict, total=False):
    messages: list[BaseMessage]      # 对话历史
    intent: str                       # 意图分类
    extracted_interview: Optional[ExtractedInterview]  # 面经提取结果
    pending_confirmation: bool        # 等待确认状态
    session_context: SessionContext   # 会话上下文
```

#### 设计评估

| 优点 | 缺点 |
|------|------|
| 意图路由延迟仅 200-500ms | Router 节点增加一次 LLM 调用 |
| 状态机清晰，易于扩展新流程 | 状态字段较多，需仔细管理 |
| 支持子图嵌套（Ingest 作为子流程） | - |

#### 后续改进

1. **Router 轻量化**：当前使用 LLM 二分类，可替换为小模型或规则引擎
2. **状态压缩**：`extracted_interview` 等大对象可在流程结束后清理
3. **并行节点**：`extract` 和 `react_loop` 可考虑并行执行

### 2.2 两阶段检索架构（召回 + 精排）

#### 为什么这样设计？

纯向量检索存在语义漂移问题：用户查询"分布式锁"可能返回"分布式事务"等语义相似但不相关的内容。CrossEncoder 可以精准建模 Query-Doc 关系，但计算代价高。

#### 怎么设计的？

```
用户查询: "Redis 分布式锁怎么实现？"
         ↓
Stage 1: 向量召回 (k*3 = 15 条候选)
         - Embedding: BGE-M3 (1024 维)
         - Qdrant ANN Search
         - 耗时: ~50ms
         ↓
Stage 2: Rerank 精排 (Top-5)
         - CrossEncoder: BGE-Reranker-Base
         - 耗时: ~100ms
         ↓
返回 Top-5 最相关题目
```

**核心代码**：

```python
# backend/app/tools/search_question_tool.py
def search_questions(query: str, company: str = None, position: str = None, k: int = 5) -> str:
    # Stage 1: 向量召回
    context = _build_query_context(query, company, position)
    query_vector = embedding_tool.embed_text(context)
    candidates = qdrant.search(query_vector, limit=k * 3)  # 召回 3 倍

    # Stage 2: Rerank 精排
    candidate_texts = [c.question_text for c in candidates]
    ranked_indices = reranker_tool.rerank(query, candidate_texts, top_k=k)

    # 根据重排结果重组
    return [candidates[idx] for idx, _ in ranked_indices]
```

#### 设计评估

| 指标 | 向量检索 | 两阶段检索 |
|------|---------|-----------|
| 召回率 | 高 | 高 |
| 精确率 | 中 | 高 |
| 延迟 | ~50ms | ~150ms |
| 资源消耗 | 低 | 中（Reranker 本地推理） |

#### 后续改进

1. **动态 k 值**：根据召回候选分数动态调整精排数量
2. **Reranker 批量化**：当前逐条精排，可批量化提升吞吐
3. **多路召回**：增加关键词召回（BM25）与向量召回融合

### 2.3 分类熔断机制

### 2.5 Redis 缓存三防设计

#### 为什么这样设计？

高并发场景下缓存面临三大问题：
1. **缓存穿透**：大量请求查询不存在的 key，绕过缓存直达数据库
2. **缓存击穿**：热点 key 过期瞬间，大量请求同时查询数据库
3. **缓存雪崩**：大量 key 同时过期，数据库压力骤增

#### 怎么设计的？

```python
# backend/app/services/cache_service.py

class CacheKeys:
    BASE_TTL = 300        # 5 分钟基础 TTL
    RANDOM_RANGE = 60     # TTL 随机范围 ±1 分钟
    NULL_MARKER = "__NULL__"  # 空值标记
    NULL_TTL = 60         # 空值缓存 1 分钟
    LOCK_TTL = 10         # 分布式锁 10 秒

    @classmethod
    def get_ttl(cls) -> int:
        """随机化 TTL，防止缓存雪崩"""
        return cls.BASE_TTL + random.randint(-cls.RANDOM_RANGE, cls.RANDOM_RANGE)
```

**防穿透**：

```python
def get(self, key: str) -> tuple[bool, Optional[Any]]:
    value = self.redis.get(key)
    if value is None:
        return (False, None)  # 缓存不存在
    if value == CacheKeys.NULL_MARKER:
        return (True, None)   # 缓存命中空值标记
    return (True, json.loads(value))  # 正常命中
```

**防击穿**：

```python
def get_with_lock(self, key: str, fetch_fn: Callable, max_retries: int = 10) -> Any:
    # 1. 查缓存
    hit, cached = self.get(key)
    if hit:
        return cached

    # 2. 尝试获取分布式锁
    for attempt in range(max_retries):
        lock_value = self._acquire_lock(key)
        if lock_value:
            try:
                # 3. 双重检查
                hit, cached = self.get(key)
                if hit:
                    return cached
                # 4. 查数据库
                data = fetch_fn()
                # 5. 写缓存
                self.set(key, data)
                return data
            finally:
                self._release_lock(key, lock_value)
        else:
            # 等待锁释放
            time.sleep(min(0.1 * (2 ** attempt), 1.0))
```

**延迟双删**：

```python
async def invalidate_question_delayed(self, question_id: str = None):
    """解决并发读写问题"""
    self.invalidate_question(question_id)  # 第一次删除
    await asyncio.sleep(1)                  # 延迟 1 秒
    self.invalidate_question(question_id)  # 第二次删除
```

#### 设计评估

| 防护机制 | 实现方式 | 效果 |
|---------|---------|------|
| 防穿透 | 缓存空值标记 | 避免无效请求直达数据库 |
| 防击穿 | 分布式锁 + 双重检查 | 避免热点 key 过期瞬间压力 |
| 防雪崩 | TTL 随机化 | 避免批量 key 同时过期 |

#### 后续改进

1. **布隆过滤器**：对大量空值场景，可用布隆过滤器预判
2. **本地缓存**：增加一层本地缓存（如 LRU）减少 Redis 压力
3. **SCAN 替代 KEYS**：大规模 key 删除时使用 SCAN 避免 Redis 阻塞

### 2.6 RabbitMQ 消费者断路器机制

#### 为什么这样设计？

Worker 消费消息后调用 LLM 生成答案，如果 LLM 服务不可用：
1. 消费者会持续失败
2. 消息不断重新入队
3. 整个系统资源被耗尽

#### 怎么设计的？

```python
# backend/app/mq/consumer.py

class AsyncRabbitMQConsumer:
    def __init__(self, prefetch_count: int = 5):
        self.circuit_breaker = create_circuit_breaker(
            fail_max=5,           # 连续失败 5 次触发熔断
            timeout_duration=30.0,  # 30 秒后尝试恢复
        )

    async def _on_message(self, message, callback):
        # 1. 熔断器拦截
        if isinstance(self.circuit_breaker.state, CircuitOpenState):
            if time.time() - self._circuit_open_time >= self._recovery_timeout:
                self.circuit_breaker.close()  # 尝试恢复
            else:
                await message.reject(requeue=True)  # 正在熔断，拒绝消息
                return

        try:
            success = await callback(task)
            if success:
                await message.ack()
                self.circuit_breaker.close()  # 成功则关闭熔断
            else:
                # 业务失败：打开熔断器，重新投递到队尾
                self.circuit_breaker.open()
                await message.ack()
                await self.republish_to_back(message, question_id, retry_count)
        except Exception as e:
            self.circuit_breaker.open()
            await message.reject(requeue=True)
```

**状态机**：

```
CLOSED (正常) ──失败达阈值──→ OPEN (熔断)
     ↑                           │
     │                           │ 30秒后
     │                           ↓
     └───成功───── HALF-OPEN (半开)
```

#### 设计评估

| 优点 | 缺点 |
|------|------|
| 防止级联故障 | 恢复时间需根据 LLM 服务特性调优 |
| 保护下游服务 | - |
| 自动恢复机制 | - |

#### 后续改进

1. **分级熔断**：对不同错误类型（网络超时 vs 业务错误）采用不同策略
2. **监控告警**：熔断器状态变更时触发告警

## 三、技术难点深度解析

### 3.1 LangGraph Checkpointer 与 PostgreSQL 持久化

#### 问题

LangGraph 的 `AsyncPostgresSaver` 使用 `psycopg3` 异步驱动，而项目中现有的 `PostgresClient` 使用 `psycopg2` 同步驱动。两者如何共存？

#### 解决方案

```python
# backend/app/db/checkpointer.py
@asynccontextmanager
async def get_checkpointer() -> AsyncGenerator[AsyncPostgresSaver, None]:
    settings = get_settings()
    db_uri = settings.postgres_url

    # pipeline=False 避免 psycopg3 generator 退出时的连接清理问题
    async with AsyncPostgresSaver.from_conn_string(db_uri, pipeline=False) as checkpointer:
        await checkpointer.setup()  # 初始化表结构
        yield checkpointer
```

**关键点**：
1. `pipeline=False`：避免 `psycopg3` 与 `asyncio` 的兼容性问题
2. 使用异步上下文管理器确保连接正确释放
3. Checkpointer 创建的表（`checkpoints`, `checkpoint_blobs`, `checkpoint_writes`）与现有表共存

### 3.2 Qdrant Payload 过滤与向量检索的平衡

#### 问题

Qdrant 支持在向量检索前进行 Payload 过滤（Pre-filtering），但如何平衡过滤条件和召回数量？

#### 解决方案

### 3.3 消息裁剪策略与上下文窗口管理

#### 问题

长对话会消耗大量 Token，如何裁剪历史消息？

#### 解决方案

```python
# backend/app/agents/graph/nodes.py
def _apply_message_trimming(messages, max_messages=10, max_tokens=8000):
    # 1. 分离 System Message
    system_messages = [m for m in messages if m.type == "system"]
    other_messages = [m for m in messages if m.type != "system"]

    # 2. 保留最近 N 条
    trimmed = other_messages[-max_messages:]

    # 3. System Message 放最前面
    if system_messages:
        trimmed = [system_messages[0]] + trimmed

    # 4. Token 估算
    estimated_tokens = sum(len(m.content) // 3 for m in trimmed)
    if estimated_tokens > max_tokens:
        # 进一步裁剪
        trimmed = trimmed[-len(trimmed) // 2:]

    return trimmed
```

**策略**：
1. **保护 System Prompt**：始终保留，放在最前面
2. **滑动窗口**：保留最近 N 条消息
3. **Token 兜底**：如果 Token 超限，进一步裁剪

## 四、面试高频问题解答

### Q1: 为什么选择 LangGraph 而不是直接用 LangChain Agent？

**回答要点**：

1. **状态管理**：LangGraph 的 `StateGraph` + `Checkpointer` 提供了原生状态持久化，而 LangChain Agent 需要手动管理
2. **流程控制**：LangGraph 的条件边和子图支持复杂的业务流程编排
3. **流式输出**：LangGraph 的 `astream_events` 可以精确捕获每个节点的 Token 流
4. **调试体验**：LangGraph Studio 提供可视化调试能力

**代码对比**：

```python
# LangChain Agent
agent = create_react_agent(llm, tools)
result = agent.invoke({"messages": messages})  # 无法精细控制流程

# LangGraph
workflow = StateGraph(AgentState)
workflow.add_conditional_edges("router", route_by_intent, {...})  # 条件路由
graph = workflow.compile(checkpointer=checkpointer)  # 状态持久化
```

### Q2: 两阶段检索中，Reranker 为什么用 CrossEncoder 而不是 Bi-Encoder？

**回答要点**：

| 类型 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| Bi-Encoder | Query 和 Doc 分别编码为向量，计算余弦相似度 | 适合大规模检索，向量可预计算 | 无法建模 Query-Doc 交互 |
| CrossEncoder | Query 和 Doc 拼接后输入模型，输出相关性分数 | 精度高，建模交互信息 | 计算代价高，不适合大规模检索 |

**结论**：两阶段检索结合两者优势——Bi-Encoder 用于大规模召回，CrossEncoder 用于小规模精排。

### Q3: 如何保证缓存与数据库的一致性？

**回答要点**：

**问题**：并发读写场景下，缓存可能与数据库不一致：

```
时刻 T1: 请求 A 读取缓存未命中，开始查数据库
时刻 T2: 请求 B 更新数据库
时刻 T3: 请求 B 删除缓存
时刻 T4: 请求 A 将旧数据写入缓存 ← 数据不一致！
```

**解决方案：延迟双删**

```python
async def invalidate_question_delayed(self, question_id: str = None):
    self.invalidate_question(question_id)  # 第一次删除
    await asyncio.sleep(1)                  # 延迟 1 秒
    self.invalidate_question(question_id)  # 第二次删除
```

**原理**：
1. 第一次删除：确保后续请求不会读到旧缓存
2. 延迟 1 秒：等待 T4 时刻的写缓存操作完成
3. 第二次删除：清除可能的脏数据

### Q4: 如何设计题目 ID 实现幂等入库？

**回答要点**：

**问题**：同一道题目可能被多次导入，如何避免重复？

**解决方案**：

```python
# backend/app/utils/hasher.py
def generate_question_id(company: str, question_text: str) -> str:
    """生成 MD5 哈希作为题目 ID"""
    combined = f"{company}_{question_text}"
    return hashlib.md5(combined.encode('utf-8')).hexdigest()
```

**Upsert 逻辑**：

```python
# backend/app/pipelines/ingestion.py
def process(self, interview: ExtractedInterview):
    for question in interview.questions:
        # 检查是否已存在
        existing = qdrant.get_question(question.question_id)
        if existing and existing.question_answer:
            # 已有答案，复用
            question.requires_async_answer = False
            payload.question_answer = existing.question_answer
        elif existing:
            # 无答案，跳过（MQ 中有待处理任务）
            continue
        else:
            # 新题目，入库
            qdrant.upsert_question(payload, vector)
```

**优点**：
1. 同一题目多次导入不会重复存储
2. 已有答案可复用，避免重复调用 LLM

### Q5: 如何实现流式输出？

**回答要点**：

**架构**：

```
用户请求 → FastAPI StreamingResponse (SSE)
         → LangGraph astream_events
         → 捕获 on_chat_model_stream 事件
         → yield Token
```

**核心代码**：

```python
# backend/app/agents/graph/workflow.py
async def astream_workflow(messages, thread_id, user_id):
    async with get_checkpointer() as checkpointer:
        workflow = create_workflow(checkpointer=checkpointer)

        async for event in workflow.astream_events(initial_state, config, version="v2"):
            kind = event.get("event")

            # 捕获 LLM Token 流
            if kind == "on_chat_model_stream":
                chunk = event["data"]["chunk"]
                if chunk and chunk.content:
                    yield {
                        "type": "token",
                        "content": chunk.content,
                    }
```
