# Offer-Catcher 多跳推理场景分析

本文档分析 Offer-Catcher Agent 中可能涉及多跳推理的业务场景，以及记忆检索策略的决策依据。

---

## 一、多跳推理场景分类

### 1.1 场景 1：跨主题关联查询

```
用户："字节跳动的 Agent 面试题和 RAG 相关的题目有什么共同考点？"

推理链：
  第1跳：检索字节跳动 Agent 面试题 → 获得题目列表
  第2跳：检索 RAG 相关题目 → 获得题目列表
  第3跳：分析共同考点 → 调用知识图谱或 LLM 总结
  
需要的能力：
  - 多次工具调用（search_questions）
  - 跨结果集分析
  - 可能需要 query_graph 查考频关系
```

### 1.2 场景 2：时间跨度回溯 + 推理

```
用户："我上周问的分布式系统问题，和今天讨论的一致性协议有什么关联？"

推理链：
  第1跳：回溯上周对话 → 从 ChatInteraction 检索
  第2跳：提取分布式系统相关内容
  第3跳：与今天的一致性协议内容对比分析
  
需要的能力：
  - 按时间范围检索 ChatInteraction
  - 提取特定话题内容
  - 跨时间对比分析
```

### 1.3 场景 3：个性化学习推荐

```
用户："根据我过去三个月的刷题记录，推荐我今天应该复习什么？"

推理链：
  第1跳：检索过去三个月 ChatInteraction → 获得刷题历史
  第2跳：分析薄弱知识点 → 从错题、重复查询中推断
  第3跳：结合 LearnedUserBehaviors → 获取兴趣偏好
  第4跳：生成复习推荐 → LLM 综合

需要的能力：
  - 时间范围检索
  - 学习进度分析（从交互中推断）
  - 结合用户行为偏好
  - 综合推荐
```

### 1.4 场景 4：知识图谱 + 用户历史结合

```
用户："RAG 和 LangChain 常一起考，那 Agent 和什么知识点关联度高？"

推理链：
  第1跳：确认 RAG-LangChain 关联（已知事实）
  第2跳：调用 query_graph 查 Agent 关联节点
  第3跳：结合用户历史查询 → 过滤不感兴趣的话题
  第4跳：返回最相关的关联知识点

需要的能力：
  - 知识图谱查询
  - 用户行为偏好检索
  - 结果过滤排序
```

### 1.5 场景 5：反馈历史影响当前响应

```
用户："上次我觉得 RAG 解释太简单，今天的 Agent 内容可以详细一点吗？"

推理链：
  第1跳：检索上次 RAG 相关 ChatInteraction
  第2跳：检索 LearnedUserFeedback → 查具体反馈内容
  第3跳：调整当前响应策略 → 应用新的 explanation_depth
  
需要的能力：
  - 关联特定话题的历史交互
  - 检索用户反馈记录
  - 动态调整响应策略
```

## 二、多跳推理类型分类

| 类型 | 描述 | 是否需要 Agent 检索记忆 | 处理方式 |
|------|------|------------------------|----------|
| **类型 A：工具链式调用** | 多次调用 search_questions、query_graph | ❌ 不需要 | react_loop 的 Agent 已处理（create_agent） |
| **类型 B：时间回溯** | 需要检索特定时间段的 ChatInteraction | ⚠️ 可选 | 简单时间查询可用规则匹配，复杂需 Agent |
| **类型 C：用户偏好结合** | 需要结合 LearnedUserBehaviors 和当前查询 | ⚠️ 可选 | LearnedUserBehaviors 可向量检索 |
| **类型 D：反馈影响响应** | 需要检索 LearnedUserFeedback 并应用 | ⚠️ 可选 | LearnedUserFeedback 可关键词匹配 |

---

## 三、关键洞察

### 3.1 类型 A 已被 react_loop 覆盖

**当前 react_loop 的 Agent 已经能处理类型 A**：

```
用户："查字节跳动 Agent 题和 RAG 题的共同点"

Agent 执行流程：
  1. 调用 search_questions(query="字节跳动 Agent")
  2. 调用 search_questions(query="RAG")
  3. LLM 分析两个结果集
  4. 总结共同考点返回给用户

这个过程完全在 react_loop 内完成，无需额外的记忆检索 Agent。
```

**原因**：`create_agent` 创建的 ReAct Agent 具备自主工具调用能力，可以根据用户意图决定调用哪些工具、调用多少次。

### 3.2 真正需要记忆检索的是类型 B/C/D

| 场景 | 需要检索的记忆类型 | 检索复杂度 |
|------|------------------|-----------|
| "上周我问的分布式系统问题" | ChatInteraction（按时间范围） | 中等（时间过滤 + 话题匹配） |
| "过去三个月的刷题记录" | ChatInteraction（按时间范围） | 中等（时间过滤） |
| "根据我的偏好推荐" | LearnedUserBehaviors | 低（向量相似度） |
| "上次我觉得 RAG 解释太简单" | LearnedUserFeedback | 低（关键词匹配） |

## 四、记忆检索策略决策

### 4.1 检索方式对比

| 方面 | Agent 检索 | 直接检索（向量/关键词） |
|------|-----------|----------------------|
| **理解能力** | ✅ 可理解复杂查询意图 | ⚠️ 基于关键词/向量相似度 |
| **检索精度** | ✅ 可判断哪些记忆真正相关 | ⚠️ 可能检索到不相关记忆 |
| **成本** | ❌ 需要额外 LLM 调用 | ✅ 无额外 LLM 成本 |
| **延迟** | ❌ 需等待 Agent 完成 | ✅ 毫秒级返回 |
| **灵活性** | ✅ 可组合多种检索策略 | ⚠️ 固定检索逻辑 |

### 4.2 场景匹配建议

| 场景 | 建议检索方式 | 理由 |
|------|-------------|------|
| **时间范围查询**（"上周"、"过去三个月"） | 规则匹配 + 向量相似度 | 时间条件可直接过滤，话题匹配用向量 |
| **用户偏好结合** | 向量相似度检索 LearnedUserBehaviors | 偏好检索相对简单，向量足够 |
| **反馈历史检索** | 关键词匹配 + 向量相似度 | 反馈内容通常有明确关键词 |
| **模糊表达**（"上次那个问题"） | Agent 检索 | 需要理解用户指的是哪个 |

### 4.3 最终结论

**在 Offer-Catcher 的场景下**：

| 检索场景 | 是否需要 Agent 检索记忆 | 推荐方式 |
|---------|------------------------|----------|
| **工具链式调用**（类型 A） | ❌ 不需要 | react_loop 的 Agent 已处理 |
| **时间范围回溯**（类型 B） | ⚠️ 初期不需要 | 规则匹配时间 + 向量相似度 |
| **用户偏好结合**（类型 C） | ⚠️ 初期不需要 | 向量相似度检索 |
| **反馈影响响应**（类型 D） | ⚠️ 初期不需要 | 关键词匹配 |
| **模糊表达理解** | ✅ 需要（后期） | Agent 检索 |

## 五、实施建议

### 5.1 分阶段策略

| 阶段 | 检索策略 | 适用场景 |
|------|---------|---------|
| **Phase 1-2** | 向量相似度 + 三维评分 | 所有记忆检索场景 |
| **Phase 3-4** | 添加规则匹配（时间过滤） | 类型 B 时间回溯场景 |
| **Phase 5** | 添加 Agent 检索 | 模糊表达、复杂历史分析 |

### 5.2 初期不使用 Agent 检索的原因

1. **ChatInteraction 的检索场景相对简单**：主要是"找相似的历史交互"，向量相似度 + 三维评分已足够
2. **Offer-Catcher 已有 embedding 工具**：`get_embeddings` 可直接使用
3. **react_loop 已处理复杂推理**：类型 A 的多跳推理不需要额外记忆检索 Agent
4. **降低成本和延迟**：Agent 检索需要额外 LLM 调用，初期优先低成本方案

### 5.3 后期引入 Agent 检索的触发条件

当以下情况出现时，考虑引入 Agent 检索：

1. **用户表达模糊**：如"上次那个问题"、"之前讨论的东西"
2. **向量检索置信度低**：相似度 < 0.5 时触发 Agent 重新检索
3. **复杂历史分析**：如"分析我过去三个月的学习趋势"
4. **多源记忆整合**：需要同时检索 ChatInteraction + LearnedUserBehaviors + LearnedUserFeedback 并整合

## 六、检索流程设计

### 6.1 标准检索流程（Phase 1-2）

```
用户输入 → Agent 响应 → 对话结束
     │
     ↓
触发异步记忆检索（后台 Worker）
     │
     ├─ 1. 提取查询 embedding
     │
     ├─ 2. 计算 ChatInteraction 三维评分
     │     score = 0.33 × recency + 0.33 × importance + 0.33 × relevance
     │
     ├─ 3. 规则匹配时间范围（如果有）
     │     如 "上周" → filter timestamp > 7 days ago
     │
     ├─ 4. 返回 Top-K 相关记忆
     │
     ↓
下次用户输入时加载到上下文
```

### 6.2 Agent 检索流程（Phase 5）

```
用户输入 → 检测是否需要 Agent 检索
     │
     ├─ 条件 1：表达模糊（"上次那个问题"）
     │
     ├─ 条件 2：向量检索置信度 < 0.5
     │
     ├─ 条件 3：复杂历史分析请求
     │
     ↓
触发 MemoryRetrievalAgent（异步）
     │
     ├─ 1. Agent 分析用户意图
     │
     ├─ 2. Agent 决定检索策略
     │     - 检索哪些记忆类型
     │     - 使用什么过滤条件
     │     - 如何整合多源结果
     │
     ├─ 3. Agent 执行检索
     │
     ├─ 4. Agent 筛选真正相关记忆
     │
     ↓
下次用户输入时加载到上下文
```

## 七、与 Claude Code 的对比

| 方面 | Claude Code | Offer-Catcher（建议） |
|------|-------------|---------------------|
| **检索触发** | 用户输入完整句子时异步触发 | 用户输入时异步触发 |
| **检索方式** | Agent 智能检索 | 初期：向量相似度；后期：Agent |
| **加载时机** | 下一次用户输入时加载 | 下一次用户输入时加载 |
| **多跳推理** | Agent 自主处理 | react_loop 的 Agent 已处理（类型 A） |
| **记忆检索 Agent** | 有专门的记忆检索 Agent | 后期引入（Phase 5） |

---

## 八、总结

Offer-Catcher 的多跳推理场景主要分为两类：

1. **工具链式调用（类型 A）**：已被 react_loop 的 Agent 覆盖，无需额外记忆检索 Agent
2. **历史记忆检索（类型 B/C/D）**：初期可用向量相似度 + 三维评分，后期根据需要引入 Agent 检索

**核心结论**：当前场景下，复杂多跳推理主要是工具链式调用，这已被现有 react_loop 覆盖。需要记忆检索的场景相对简单，向量相似度检索初期足够。
