## 一、现状分析

### 1.1 当前架构

```
┌─────────────────────────────────────────────────────────────────┐
│                     ChatAgent (内存层)                           │
│  _session_state: dict[str, dict]                                │
│  - intent, params, extracted_interview                          │
│  - pending_confirmation, confirmed_data                         │
│  - current_subgraph, last_tool_result, context                  │

└────────────────────────────┬────────────────────────────────────┘
                             │ 每次调用后从 final_state 更新
┌────────────────────────────▼────────────────────────────────────┐
│              LangGraph Checkpointer (PostgreSQL)                 │
│  AsyncPostgresSaver.from_conn_string()                          │
│  表: checkpoints, checkpoint_blobs, checkpoint_writes           │
│  ✅ 自动恢复工作流状态（通过 thread_id）                           │
└────────────────────────────┬────────────────────────────────────┘
                             │ 独立存储
┌────────────────────────────▼────────────────────────────────────┐
│              PostgresClient (对话历史层) - 未使用                  │

│  ⚠️ 问题：ChatAgent 未调用，前端历史无法持久化                     │

```

**当前 Chat API 调用流程：**

```python

async def chat_stream(request: ChatRequest):
    # 1. 从请求中获取前端传来的历史消息

```

**前端状态管理：**


### 1.3 核心问题

| 问题 | 影响 | 优先级 |
|-----|------|-------|
| 1. 内存状态 `_session_state` 易丢失 | 服务重启后需要用户重新操作 | 中 |
| 2. 前端传入历史消息，非服务端加载 | 无法跨设备同步，历史无法持久化 | 高 |
| 3. `session_id` 每次刷新生成 | 无法恢复之前的会话 | 高 |
| 4. Checkpointer 与 PostgresClient 功能重叠 | 双重存储，增加复杂度 | 中 |
| 5. 无历史会话管理 UI | 用户体验差 | 高 |

### 2.2 核心改动

#### 改动一：移除 ChatAgent 内存状态

**原理：** LangGraph Checkpointer 已提供完整的状态恢复能力。

**当前实现问题：**
```python
# chat_agent.py
class ChatAgent:
    def __init__(self):
        self._session_state: dict[str, dict] = {}  # 内存存储

    async def achat_streaming(self, message, history, session_id):
        session_state = self._get_session_state(session_id)  # 从内存取
        # ... 运行工作流 ...
        self._update_session_state(session_id, result)  # 更新内存
```

**优化后：**
```python
# chat_agent.py (优化版)
class ChatAgent:
    async def achat_streaming(self, message, conversation_id):
        # LangGraph Checkpointer 自动通过 conversation_id 恢复状态
        # 不再需要手动管理 _session_state
        async for event in astream_workflow(
            messages=[HumanMessage(content=message)],
            thread_id=conversation_id,  # 使用 conversation_id 作为 thread_id
        ):
            yield event
```

**状态恢复流程：**
1. Checkpointer 通过 `thread_id` 从 `checkpoints` 表加载上次状态
2. 新消息追加到 `messages` 列表
3. 工作流执行完毕后，新状态自动保存到 `checkpoints` 表

#### 改动二：统一 conversation_id 和 thread_id

**当前问题：**
- 前端用 `session_${Date.now()}` 作为 session_id
- LangGraph Checkpointer 用 thread_id
- PostgresClient 用 conversation_id
- 三者概念相同但未统一

**优化方案：**
```python
# 统一使用 conversation_id
conversation_id = uuid.uuid4()  # 创建会话时生成

# LangGraph Checkpointer
thread_id = conversation_id  # 直接复用

# PostgresClient
conversation_id = conversation_id  # 保持一致
```

#### 改动三：服务端管理消息历史

**当前问题：**
```python
# 前端传入历史
history_messages = [msg for msg in request.history]
```

**优化后：**
```python
# 服务端从数据库加载
async def chat_stream(request: ChatRequest):
    pg = get_postgres_client()

    # 加载历史消息（不传 history 参数）
    db_messages = pg.get_messages(user_id, request.conversation_id)

    # 转换为 LangChain 消息
    history = [
        HumanMessage(content=m.content) if m.role == "user"
        else AIMessage(content=m.content)
        for m in db_messages
    ]

    # 运行工作流
    async for chunk in agent.achat_streaming(
        message=request.message,
        history=history,
        conversation_id=request.conversation_id
    ):
        yield chunk

    # 保存新消息
    pg.add_message(user_id, conversation_id, "user", request.message)
    pg.add_message(user_id, conversation_id, "assistant", response)
```

### 3.3 数据模型

#### PostgreSQL 表结构（已有，无需修改）

```sql
-- 会话表
CREATE TABLE conversations (
    id VARCHAR(36) PRIMARY KEY,
    user_id VARCHAR(36) NOT NULL,
    title VARCHAR(500) NOT NULL,
    created_at TIMESTAMP NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMP NOT NULL DEFAULT NOW()

CREATE TABLE messages (
    id VARCHAR(36) PRIMARY KEY,

);
```

## 五、风险与应对

| 风险 | 影响 | 应对措施 |
|-----|------|---------|
| LangGraph Checkpointer 与 PostgresClient 事务不一致 | 状态不一致 | 使用同一个 PostgreSQL 连接，事务内完成所有操作 |
| 大量历史消息导致加载慢 | 性能问题 | 分页加载，仅加载最近 N 条消息 |
| 并发写入冲突 | 数据丢失 | 使用 PostgreSQL 行锁 |
