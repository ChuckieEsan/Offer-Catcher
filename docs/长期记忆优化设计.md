# 长期记忆优化设计文档

## 1. 背景与现状

### 1.1 当前记忆架构

项目使用 LangGraph 提供的两层记忆架构：

| 层级 | 组件 | 存储内容 | 持久化 |
|------|------|---------|--------|
| 短期记忆 | `AsyncPostgresSaver` (Checkpointer) | 对话历史、意图、参数 | 按 thread_id 自动保存恢复 |
| 长期记忆 | `PostgresStore` (Store) | 用户画像、偏好、学习进度 | 跨会话共享 |

### 1.2 存在的问题

| 问题 | 现状 | 影响 |
|------|------|------|
| PostgresStore 未启用向量检索 | 只有 `put/get/search` 基础功能 | 无法语义化查找相关记忆 |
| 无会话摘要生成 | `SessionSummary` 定义了但未实际使用 | 历史对话无法有效利用 |
| 记忆检索靠精确匹配 | 只能按 namespace + key 查询 | 无法找到语义相关的历史 |

## 2. 改进方案架构

### 2.1 整体架构

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                           对话结束时触发                                      │
│  ┌─────────────────────────────────────────────────────────────────────┐   │

│  │  输入：对话历史 → 输出：结构化摘要                                      │   │
│  │  - summary: 摘要文本（100-200字）                                     │   │
│  │  - key_topics: 关键话题（3-5个）                                      │   │
│  │  - entities_mentioned: 提及的知识点/公司                              │   │
│  │  - questions_discussed: 讨论过的题目ID                               │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────────┐
│                     PostgresStore (启用向量检索)                             │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  Namespace: ("users", user_id, "episodes")                          │   │
│  │  每条记录自动生成 Embedding，支持语义检索                               │   │
│  │  向量字段: summary, key_topics                                       │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  Namespace: ("users", user_id, "profile")     ← 现有，保持不变        │   │
│  │  Namespace: ("users", user_id, "preferences") ← 现有，保持不变        │   │

│  └─────────────────────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────────────────────┘
                                      │
                                      ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│                        对话开始时检索                                        │
│  ┌─────────────────────────────────────────────────────────────────────┐   │

│  │      namespace=("users", user_id, "episodes"),                      │   │

│  │      limit=3                                                        │   │
│  │  )                                                                  │   │
│  │  → 返回语义相关的历史会话摘要（约 600-900 tokens）                      │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 2.2 记忆类型对照

根据 LangGraph 最新文档，记忆分为三种类型：

| 类型 | 说明 | 本项目实现 |
|------|------|-----------|
| **Episodic（情景记忆）** | 过去对话的记录 | `EpisodeMemory`（本次新增） |
| **Semantic（语义记忆）** | 事实、偏好、知识 | `UserProfile`, `UserPreferences`（现有） |
| **Procedural（过程记忆）** | 技能、操作方法 | 暂未实现，可后续扩展 |

## 3. 摘要大小设计

### 3.1 设计决策因素

| 因素 | 影响 |
|------|------|
| **向量检索质量** | 太短语义信息不足，太长噪音稀释关键信息 |
| **Token 消耗** | 每次对话注入 N 条摘要，累计成本 |
| **LLM 生成成本** | 生成摘要本身的 Token 消耗 |

### 3.2 推荐配置

| 字段 | 推荐大小 | 说明 |
|------|---------|------|
| `summary` | **100-200 字** | 核心摘要，会被向量化 |
| `key_topics` | 3-5 个关键词 | 话题标签，辅助向量检索 |
| `entities_mentioned` | 5-10 个 | 知识点/公司，精确匹配用 |
| `questions_discussed` | 题目 ID 列表（最多 5 个） | 关联题目，不做向量化 |

### 3.3 大小决策理由

```
太短（< 50 字）：
  "讨论了 Redis 分布式锁。"
  → 丢失细节，无法区分不同会话

适中（100-200 字）：
  "讨论了 Redis 分布式锁的实现方案，包括 setnx、Redlock 和 watch 机制。
   用户对 Redlock 在极端情况下的安全性有疑问，建议深入了解 CAP 理论。"
  → 保留关键信息，语义清晰

太长（> 400 字）：
  包含完整对话细节 → 噪音多，向量检索效果下降
```

### 3.4 Token 预算计算

假设每次对话注入 **3 条历史摘要**：

| 项目 | 计算 |
|------|------|
| summary 字段 | 3 × 200 字 ≈ 150 tokens |
| key_topics | 3 × 5 个词 ≈ 30 tokens |
| entities | 3 × 10 个 ≈ 20 tokens |
| **总计** | **~200 tokens** |

对比直接注入历史对话：
- 20 条历史消息 ≈ 2000-5000 tokens

**结论：节省 90%+ 的上下文空间**

### 3.5 不同场景的调整建议

| 场景 | 摘要大小 | 原因 |
|------|---------|------|
| 面经解析 | 50-100 字 | 主要是提取题目，信息密度低 |
| 题目练习 | 150-200 字 | 需要记录用户表现和薄弱点 |
| 深度技术讨论 | 200-300 字 | 技术细节多，需要保留上下文 |
| 闲聊 | 不生成摘要 | 无需记忆 |

## 4. 核心数据模型

### 4.1 EpisodeMemory

```python
class EpisodeMemory(BaseModel):
    """会话情景记忆（支持向量检索）

    Attributes:
        session_id: 会话唯一标识
        summary: 摘要文本（100-200字），会被向量化
        key_topics: 关键话题列表（3-5个），会被向量化
        entities_mentioned: 提及的知识点/公司列表
        questions_discussed: 讨论过的题目ID列表
        created_at: 创建时间
    """
    session_id: str
    summary: str
    key_topics: list[str] = Field(default_factory=list)
    entities_mentioned: list[str] = Field(default_factory=list)
    questions_discussed: list[str] = Field(default_factory=list)
    created_at: str = Field(default_factory=lambda: datetime.now().isoformat())
```

### 4.2 存储结构

```
PostgresStore
├── ("users", user_id, "profile")          → UserProfile（现有）
├── ("users", user_id, "preferences")      → UserPreferences（现有）
├── ("users", user_id, "progress")         → LearningProgress（现有）
└── ("users", user_id, "episodes")         → EpisodeMemory（新增）
     └── key: session_id
     └── value: EpisodeMemory.model_dump()
     └── 向量索引: summary, key_topics
```

```python
index_config = IndexConfig(
    embed=_get_embedding_function(),  # 使用 BGE-M3

    fields=["summary", "key_topics"],  # 只对摘要和话题建索引
)
```

## 8. 预期效果

### 8.1 用户体验示例

```
用户: "上次我们讨论的分布式锁的题目是什么？"

Agent 内部流程:
1. 向量检索 store.search(query="分布式锁", limit=3)
2. 找到 2 周前的会话摘要:
   - "讨论了 Redis 分布式锁的实现，包括 setnx 和 Redlock 方案"
   - "用户对 Redlock 的安全性有疑问"
3. 结合检索结果回答: "上次我们讨论的是 Redis 分布式锁的实现方式..."

用户: "我最近在准备字节跳动的面试，有什么建议？"

Agent 内部流程:
1. 向量检索 store.search(query="字节跳动 面试", limit=3)
2. 找到相关历史:
   - "用户之前练习过字节跳动的 Kafka 相关题目"
   - "用户对分布式系统有基础了解"
3. 个性化推荐: "根据你之前的练习，建议重点复习 Kafka 消费者组..."
```

### 8.2 性能指标

| 指标 | 优化前 | 优化后 |
|------|--------|--------|
| 历史上下文注入 | 2000-5000 tokens | 200-600 tokens |
| 记忆检索方式 | 精确匹配 | 语义检索 |
| 跨会话记忆利用 | 无 | 支持检索相关会话 |
