## 一、评测框架总览

基于已有的评测设计文档（`基于 Harness Engineering 的记忆评测方案.md`），结合当前实际代码状态和问题，制定可执行的评测方案。

评测分为 **四个核心 Harness + 工程稳健性测试**：

| Harness | 核心问题 | 当前状态 | 优先级 |
|---------|---------|---------|--------|
| **Extraction** | 该记的记住了吗？不该记的跳过了吗？ | 50% 通过率，需优先修复 | P0 |
| **Retrieval** | 能在需要时召回正确记忆吗？ | 未系统性评测 | P1 |
| **Utilization** | 召回的记忆真正改善回答了吗？ | 未评测 | P1 |
| **Systems** | 异步/并发/故障下稳健吗？ | 有锁机制但未压力测试 | P2 |

## 二、Extraction Harness 评测方案

### 2.1 评测目标

评测系统是否能够从对话中正确提取：

- 值得长期记忆的偏好
- 值得长期记忆的行为模式
- 值得摘要存储的历史事件
- **不应进入长期记忆的临时信息**

### 2.2 测试场景分类

#### 必测场景矩阵（建议扩充至 30+ 条）

| 类别 | 场景 | 预期行为 | 当前测试覆盖 |
|------|------|----------|-------------|
| **显式偏好** | "以后都用中文回答" | write_preferences + index | ✓ |
| **弱显式偏好** | "这类问题最好多给例子" | write_preferences（低权重） | 缺失 |
| **负向反馈** | "不要这么绝对" | write_preferences（负向部分） | ✓ |
| **临时约束** | "这次先简短一点" | **只 cursor**，不写长期记忆 | **缺失（关键）** |
| **单次行为** | 一次"原理→实现"序列 | **只 cursor**，需2次确认 | ✓（但失败） |
| **重复行为** | 两次以上相同序列 | write_behaviors + index | ✓（但失败） |
| **技术讨论** | 3轮以上深度问答 | write_session_summary | ✓ |
| **闲聊/确认** | "好的谢谢" | **只 cursor** | ✓ |
| **偏好冲突** | 新偏好覆盖旧偏好 | 替换而非追加 | ✓（timeout） |
| **偏好重复** | 与已有偏好语义相同 | **只 cursor**，去重 | ✓（但失败） |
| **摘要重复** | 继续讨论同一话题 | **只 cursor**，去重 | ✓（但失败） |

## 四、Utilization Harness 评测方案

### 4.1 评测目标

评测"记忆被取回后，是否真正提升了最终回答"。

**这是最关键的评测**，因为它反映用户感知价值。

### 4.2 对照实验设计

对同一 case 运行三组实验：

| 组别 | 配置 | 目的 |
|------|------|------|
| **No Memory** | 关闭记忆系统 | 基线 |
| **Memory Enabled** | 正常使用记忆 | 实际效果 |
| **Oracle Memory** | 直接注入 gold 记忆 | 理论上限 |

可选第四组：

| **Noise Memory** | 注入无关记忆 | 测试抗噪声能力 |

## 五、Systems Harness 评测方案

### 5.1 评测目标

评测记忆系统在真实工程环境中的稳健性：

- 异步检索
- 后台提取
- 主 Agent 直接写记忆
- 锁与并发
- 故障重试
- 容量裁剪
- 幂等与一致性

### 5.2 并发冲突测试

#### 场景：后台 worker 和主 Agent 同时更新

```python
async def test_concurrent_update_conflict():
    """后台 memory worker 正在处理时，主 Agent 直写偏好"""

    # 1. 启动后台 memory worker，开始处理旧游标
    worker_task = asyncio.create_task(
        run_memory_agent(user_id, conversation_id, messages)
    )

    # 2. 用户下一轮显式要求更新偏好，主 Agent 立即直写
    await asyncio.sleep(0.1)  # 让 worker 先开始
    main_agent_write(user_id, "preferences", "新的偏好内容")

    # 3. 等待 worker 完成
    await worker_task

    # 4. 验证：worker 不应覆盖主 Agent 的更新
    current_preferences = read_preferences(user_id)
```

## 十、高风险点清单

以下是此类记忆系统最值得重点关注的风险点，建议在 Harness 中单独设标签与回归用例：

| # | 风险点 | 描述 | 对应 Harness |
|---|--------|------|-------------|
| 1 | 临时约束误写为长期偏好 | "这次先简短"被写入 preferences | Extraction |
| 2 | 单次行为写入 behaviors | 只观察一次就写入，违反 >= 2 次规则 | Extraction |
| 3 | 同一 conversation 多条摘要被错误去重 | 按 conversation_id 去重而非内容 | Retrieval |
| 4 | 过时偏好覆盖新偏好 | 检索返回旧偏好，新偏好被忽略 | Retrieval |
| 5 | 摘要压缩导致关键细节丢失 | summary 过短，检索后无法恢复细节 | Extraction |
| 6 | 常驻概要过强，阻止按需深读 | MEMORY.md 概要已包含信息，不触发 reference 加载 | Retrieval |
| 7 | 检索成功但回答未使用 | 召回正确记忆但模型未在回答中体现 | Utilization |
| 8 | 主 Agent 直写与后台更新冲突 | 后台 worker 覆盖主 Agent 的直写 | Systems |
| 9 | 基于字符串标记的互斥协议不稳定 | cursor UUID 机制在某些边界情况失效 | Systems |
| 10 | 上下文裁剪只看顺序不看相关性 | 裁剪策略导致关键记忆丢失 | Systems |
| 11 | 去重判断不可靠 | 纯文本判断语义等价失败 | Extraction |
| 12 | 系统降级后 silent corruption | 故障恢复后数据不一致 | Systems |

## 十三、一句话原则清单

- 先测"该不该记"，再测"能不能取"，最后测"有没有用"
- 没有 trace 的评测，基本不可归因
- Retrieval 分高不代表用户体验好，Utilization 才是关键
- Systems Harness 不是附属项，而是上线前的必选项
- Oracle 实验能帮你快速定位问题是在检索链还是使用链
- 评测中要明确区分长期偏好、历史事件、临时约束
- 去重、裁剪、并发互斥，是最容易在真实系统中出问题的地方
- Extraction 的 over_memory_rate 是当前最严重的问题，优先修复
