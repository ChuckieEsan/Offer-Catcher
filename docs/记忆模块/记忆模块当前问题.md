## 三、问题根因分析

### 3.1 核心问题一：过度记忆

**现象**：Agent 在不该写入的场景仍然写入了 behaviors 或 session_summary。

**失败案例**：
- `behavior_single_instance`：单次观察就写入 behaviors（违反"至少 2 次"规则）
- 多个场景都额外写入 session_summary

**根因**：提示词中虽然定义了"至少 2 次"规则，但没有强制约束语气，LLM 可能忽略。

**提示词现状**：
```markdown
<merge_strategy>
- 新偏好与现有偏好一致时：不写入（避免重复）
...
</merge_strategy>
```

问题：规则描述过于温和，缺少强制语气和明确的否定约束。

### 3.2 核心问题二：去重能力缺失

**现象**：Agent 无法判断新内容是否与已有内容重复。

**失败案例**：
- `preference_duplicate`：用户说"回答简洁一点"，已有偏好是"偏好简洁直接"，Agent 仍然写入
- `session_summary_duplicate_topic`：追问同一话题，Agent 仍然写入 session_summary

**根因**：
1. 提示词只说"新偏好与旧偏好一致时，跳过不写"，但没有给出**判断标准**
2. Agent 收到的 context 中只有 `current_preferences` 文本，没有语义相似度判断能力
3. LLM 无法可靠地判断"简洁" ≈ "简洁直接"这种语义等价

**架构限制**：
- 当前设计依赖 LLM 的文本理解能力进行去重
- 没有 embedding 相似度检索辅助判断
- 没有结构化的偏好字段定义（难以精确比较）

### 3.3 核心问题三：工具调用边界模糊

**现象**：Agent 经常同时调用多个工具，超出必要范围。

**失败案例**：几乎所有场景都额外调用了 `write_session_summary`

**根因**：
- 提示词没有明确禁止同时调用多个记忆类型
- 没有定义"每轮对话最多写入 X 条记忆"的硬性约束
- session_summary 的触发条件过于宽松

## 四、提示词优化建议

### 4.1 增加强制约束

```markdown
<strict_rules>
**工具调用硬性约束**：

1. **behaviors 写入门槛**：
   - 必须观察到 **至少 2 次** 完全相同的模式序列
   - 单次观察 → **绝不允许** 写 behaviors
   - 单次观察可以写 session_summary 或直接跳过

2. **每轮对话限制**：
   - session_summary：最多 1 条
   - preferences：最多 1 次更新
   - behaviors：最多 1 次更新
   - 不允许同时调用多个记忆写入工具（除非 preferences + behaviors + index）

3. **update_cursor 必须最后调用**
</strict_rules>
```

### 4.2 增加去重判断指令

```markdown
<dedup_rules>
**偏好去重判断**：

比较 current_preferences 与新表达的偏好：
- 提取偏好类别（响应风格、话题偏好、语言偏好等）
- 判断意思是否相同：
  - "简洁" ≈ "简洁直接" ≈ "不要太长"
  - "详细" ≈ "深入" ≈ "完整解释"
- 如果类别相同且意思相近 → **跳过不写**

**摘要去重判断**：

检查 new_messages 是否只是延续之前的话题：
- 只是追问细节 → 跳过
- 只是确认/感谢 → 跳过
- 只是表达偏好 → 写 preferences 而非 session_summary

**判断困难时的默认行为**：
- 如果不确定是否重复 → **倾向于不写**（宁缺毋滥原则）
</dedup_rules>
```

### 4.3 增加负面约束

```markdown
<negative_constraints>
**session_summary 不应写入的场景**：

以下场景 **绝对禁止** 写 session_summary：
1. 只是追问同一话题的细节
2. 只是确认或感谢
3. 只是表达偏好（偏好应写入 preferences）
4. 只是闲聊或模糊表达
5. 对话内容少于 3 轮
6. 没有解决具体问题

**宁缺毋滥原则**：
记忆系统的价值在于精准而非数量。
写入无价值记忆会污染检索结果。
</negative_constraints>
```

## 五、架构层面建议

### 5.1 去重需要 embedding 辅助

当前纯文本判断不可靠，建议：

```python
# 伪代码
def should_write_preference(new_preference: str, current_preferences: str) -> bool:
    # 1. 提取已有偏好的 embedding
    existing_embeddings = extract_preference_embeddings(current_preferences)

    # 2. 计算新偏好的 embedding
    new_embedding = embedding_adapter.embed(new_preference)

    # 3. 相似度判断
    for existing in existing_embeddings:
        similarity = cosine_similarity(new_embedding, existing)
        if similarity > 0.85:  # 高相似度阈值
            return False  # 重复，不写入

    return True  # 不重复，写入
```

### 5.2 偏好结构化存储

当前 preferences.md 是自由格式 markdown，难以精确比较。建议使用结构化 schema：

```python
class Preference(BaseModel):
    category: PreferenceCategory  # 响应风格、话题偏好、语言偏好
    content: str  # 偏好内容
    sentiment: SentimentType  # positive / negative
    created_at: datetime
```

这样可以精确判断：
- 同一 category 的偏好是否冲突
- 同一 content 的偏好是否重复

### 5.3 记忆质量评分

增加一个评分机制，过滤低价值记忆：

```python
def evaluate_memory_value(content: str, context: dict) -> float:
    # 评分维度：
    # - 信息密度（关键词数量）
    # - 独特性（与已有记忆的相似度）
    # - 可复用性（是否是通用经验）
    # - 明确性（是否是明确的偏好表达）

    score = calculate_score(content, context)

    return score  # 0-1，低于阈值则不写入
```
