## 一、设计理念

### 1.1 核心洞察

> 记忆本质上就是用户自定义的 Skill——指导 Agent 行为的用户特定规则文档。

因此，记忆模块完全遵循 [Agent Skills 标准](https://agentskills.io/specification)：
- 主文档（MEMORY.md）+ references 目录的结构
- 渐进式披露（Progressive Disclosure）加载机制
- LLM 直接阅读、编辑、按需查询

### 1.2 与全局 Skill 的对比

| 方面 | 全局 Skill | 用户记忆（MEMORY） |
|------|-----------|-------------------|
| **范围** | 所有用户共享 | 单用户特定 |
| **位置** | `.agents/skills/` | `memory/{user_id}/` |
| **结构** | SKILL.md + references | MEMORY.md + references |
| **加载** | 按触发条件加载 | MEMORY.md 始终加载，references 按需查询 |
| **创建方式** | 开发者编写 | 用户反馈积累 / UI 创建 |

## 二、数据边界划分

### 2.1 存储分类

| 数据 | 存储位置 | 内容 | 用途 |
|------|---------|------|------|
| **mastery_level** | Qdrant | 题目掌握程度（LEVEL_0/1/2） | 客观事实，判断薄弱题目 |
| **preferences.md** | 记忆 references | 用户显式表达的偏好 | 指导响应风格 |
| **behaviors.md** | 记忆 references | 用户交互行为模式 | 指导响应策略 |
| **session_summaries** | 数据库表 | 会话摘要 + 向量 | 支持历史回溯、语义检索 |

### 2.2 核心原则

- **规则**（preferences/behaviors）→ Markdown 文件，放入 references，LLM 按需调用
- **数据**（session_summaries）→ 数据库表，支持向量检索
- **消息**（messages）→ 数据库表，完整历史，用于生成摘要

## 四、与现有模块的关系

### 4.1 继续保留的模块

| 模块 | 用途 | 与记忆的关系 |
|------|------|-------------|
| conversations | 对话元信息 | session_summaries 通过 conversation_id 关联 |
| messages | 完整消息内容 | 用于生成 summary、embedding |
| UserProfile | 用户身份信息 | 独立存在，不与记忆同步 |
| Qdrant mastery_level | 题目掌握程度 | 独立存在，不放入记忆 |

### 4.2 新增模块

| 模块 | 位置 | 用途 |
|------|------|------|
| MEMORY.md | PostgresStore | 主文档（始终加载） |
| preferences.md | PostgresStore | 用户偏好规则 |
| behaviors.md | PostgresStore | 行为模式规则 |
| skills/{skill}/ | PostgresStore | 用户自定义 Skill |
| session_summaries | 数据库表 | 会话摘要 + 向量 |

### 4.3 preferences.md 的职责定位

**核心决策**：Agent 响应偏好完全由 `preferences.md` 管理，用户通过记忆页面直接编辑。

**设计原因**：
- 避免数据同步冲突（不引入额外的 UserSettings 系统）
- 用户直接编辑 preferences.md 更直观（所见即所得）
- Agent 直接读取 preferences.md，无需通过中间层中转

### 4.4 数据流向图

```
用户对话
    ↓
messages 表（存储完整消息）
    ↓
对话结束 → 异步 Worker
    ↓
LLM 生成 → summary + embedding
    ↓
session_summaries 表（CASCADE DELETE）
    ↓
从 summary 提取反馈 → preferences.md
从对话分析 → behaviors.md
    ↓
MEMORY.md（同步更新概要）
```

## 五、MEMORY.md 主文档设计

### 5.1 格式规范

```markdown
---
name: user-memory-{user_id}
description: 用户特定的偏好和行为规则。始终加载此文档。
             当概要信息不足以回答问题时，使用 load_memory_reference Tool
             或 search_session_history Tool 查询详情。
---

# 用户记忆

## 偏好概要
- 解释深度：深入详细
- 代码示例：偏好给出代码示例
- 语言：中文

## 行为模式概要
- 提问序列：问 what 后常问 why
- 关注焦点：偏好实践应用
- 追问风格：追问具体细节

## 会话历史概要
最近 5 次会话：
- 2026-04-12 | 面试模拟复盘（字节跳动 Agent 岗，得分 65）
- 2026-04-11 | Agent 架构对比
- 2026-04-10 | RAG 原理讨论

需要更多历史详情时，使用 `search_session_history` Tool。

## 可用 References
| Reference | 描述 | 建议调用时机 |
|-----------|------|-------------|
| `preferences` | 完整的用户偏好设置 | 用户表达反馈 / 回答需要个性化调整 |
| `behaviors` | 观察到的行为模式详情 | Agent 需要调整响应策略 |

## 可用自定义 Skill
| Skill | 描述 | 触发条件 |
|-------|------|---------|
| `interview_tips` | 面试技巧 | 模拟面试开始时 |

## 使用指南
1. 本文档始终加载，提供概要信息
2. 概要不够详细时，调用 `load_memory_reference("{reference_name}")`
3. 需要语义检索历史时，调用 `search_session_history("{query}")`
4. 触发自定义 Skill 时，调用 `load_skill("{skill_name}")`
```

### 5.2 长度控制

目标：**300-500 tokens**

- 偏好概要：~50 tokens（3-5 条要点）
- 行为模式概要：~100 tokens（3-4 条要点）
- 会话历史概要：~100 tokens（最近 5 条）
- References/Skill 列表：~150 tokens

## 六、references 详情设计

### 6.1 preferences.md

```markdown
# 用户偏好详情

## 响应风格
- 语言：中文
- 解释深度：深入详细，尤其针对生疏知识点要鞭辟入里
- 响应长度：适中，但允许长文解释复杂概念
- 代码示例：偏好给出代码示例

## 话题偏好

### RAG
- 必须给出代码示例
- 优先解释实践细节（召回阈值、重排序），而非向量原理
- 涉及召回阈值时，给出具体数值范围（如 0.7-0.85）

### Agent
- 优先实现细节和错误处理
- 对比 LangChain vs LangGraph 时，给出表格形式

### Python
- 跳过基础语法解释
- 重点讲解架构设计和最佳实践

## 反馈历史
| 日期 | 反馈内容 | 影响 |
|------|---------|------|
| 2026-04-10 | "RAG 解释不够深入，缺少代码" | 添加 RAG 话题偏好：必须代码示例 |
| 2026-04-12 | "解释鞭辟入里" | 更新响应风格：解释深度为深入详细 |
```

### 6.2 behaviors.md

```markdown
# 用户行为模式详情

## 提问序列
用户经常在问"什么是 X"之后追问"为什么需要 X"或"X 怎么实现"。

**建议**：回答 what 类问题时，主动补充 why 和 how 的信息。

## 关注焦点
用户偏好 **practical_use**（实践应用）类型的答案。

**证据**：用户追问"怎么用"、"代码怎么写"的频率高于"原理是什么"。

**建议**：检索时优先返回带代码示例的题目。

## 追问风格
用户的追问通常针对 **具体细节**（参数、边界情况、错误处理）。

**建议**：回答时主动覆盖常见边界情况、错误处理方式。

## 知识背景推测
用户有一定 Python 基础，对 LangChain API 有使用经验，但对架构设计理解不深。

**建议**：可以跳过 Python 基础解释，但架构层面要详细展开。
```

## 七、session_summaries 表设计

### 7.1 设计理念

> session_summaries 是 conversations 的增强层，不是独立的会话记录。

| 表 | 职责 | 存储内容 |
|---|------|---------|
| **conversations** | 对话管理 | title, created_at, updated_at |
| **messages** | 消息存储 | 完整的对话消息内容 |
| **session_summaries** | 检索 | summary, embedding |

### 7.2 表结构

```sql
CREATE TABLE session_summaries (
    conversation_id VARCHAR(36) PRIMARY KEY,
    user_id VARCHAR(36) NOT NULL,
    summary TEXT NOT NULL,
    embedding vector(1024),
    session_type VARCHAR(20) DEFAULT 'chat',
    metadata JSONB,
    created_at TIMESTAMP NOT NULL DEFAULT NOW(),
    FOREIGN KEY (conversation_id) REFERENCES conversations(id) ON DELETE CASCADE
);

-- 索引
CREATE INDEX idx_session_summaries_user_id ON session_summaries(user_id);
CREATE INDEX idx_session_summaries_user_created ON session_summaries(user_id, created_at DESC);

-- 向量索引
CREATE INDEX idx_session_summaries_embedding ON session_summaries 
USING ivfflat (embedding vector_cosine_ops) WITH (lists = 100);
```

### 7.3 字段说明

| 字段 | 类型 | 说明 |
|------|------|------|
| conversation_id | VARCHAR(36) | 主键，关联 conversations |
| user_id | VARCHAR(36) | 用户 ID |
| summary | TEXT | 会话摘要（主题 + 关键问题 + 用户反馈） |
| embedding | vector(1024) | 语义向量（BGE-M3） |
| session_type | VARCHAR(20) | 类型：'chat' / 'interview' |
| metadata | JSONB | 元数据（面试得分、公司等） |
| created_at | TIMESTAMP | 创建时间 |

### 7.4 CASCADE DELETE

删除 conversation 时自动删除 session_summaries，保证数据一致性。

## 八、记忆生命周期

### 8.1 整体流程图

```
┌─────────────────────────────────────────────────────────────┐
│ 用户注册 / 首次对话                                            │
│                                                              │
│ 初始化：创建 MEMORY.md + preferences.md + behaviors.md        │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│ 用户发起有效提问（≥10字）                                       │
│                                                              │
│ 1. MEMORY.md 始终加载                                         │
│ 2. 触发异步 Worker 检索 session_summaries                     │
│ 3. 检索结果写入 Checkpointer（memory_context）                 │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│ 下次对话开始前                                                 │
│                                                              │
│ 1. 从 Checkpointer 读取 memory_context                        │
│ 2. 注入到 System Message                                      │
│ 3. LLM 可按需调用 Tool 加载详情                                 │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│ 对话结束                                                      │
│                                                              │
│ 异步 Worker：                                                 │
│ 1. 生成 summary + embedding → session_summaries              │
│ 2. 提取反馈 → preferences.md                                  │
│ 3. 分析行为 → behaviors.md                                    │
│ 4. 同步概要 → MEMORY.md                                        │
└─────────────────────────────────────────────────────────────┘
```

```python
async def initialize_user_memory(user_id: str) -> None:
```

### 8.3 记忆检索流程

#### 8.3.1 触发条件

- 消息长度 ≥ 10 个字符
- 非空字符串
- 每次有效提问触发检索

#### 8.3.2 异步检索 Worker

```python
async def retrieve_memory_worker(
    user_id: str,
    conversation_id: str,
    query: str,
    checkpointer: PostgresSaver,
):
    """异步检索记忆
    
    流程：
    1. 计算 query embedding
    2. 语义检索 session_summaries（top_k = 5）
    3. 获取对话标题
    4. 格式化为 memory_context
    5. 读取当前 checkpoint，更新 memory_context
    """
    settings = get_settings()
    
    try:
        # 1. 计算 embedding
        query_embedding = get_embedding_tool().embed_text(query)
        
        # 2. 语义检索
        results = pg.search_session_summaries(
            user_id=user_id,
            query_embedding=query_embedding,
            top_k=settings.memory_retrieval_top_k,
        )
        
        # 3. 获取标题
        for r in results:
            conv = pg.get_conversation(user_id, r["conversation_id"])
            if conv:
                r["title"] = conv.title
                r["created_at"] = conv.created_at
        
        # 4. 更新 checkpoint
        config = {"configurable": {"thread_id": conversation_id}}
        current_checkpoint = await checkpointer.aget(config)
        
        if current_checkpoint:
            current_memory = current_checkpoint.channel_values.get("memory_context", "")
            injected_ids = current_checkpoint.channel_values.get("injected_session_ids", [])
            
            merged_context, merged_ids = merge_memory_context(
                current_memory, injected_ids, results
            )
            
            current_checkpoint.channel_values["memory_context"] = merged_context
            current_checkpoint.channel_values["injected_session_ids"] = merged_ids
            
            await checkpointer.aput(config, current_checkpoint)
        
    finally:
        await release_memory_retrieval_lock(user_id, conversation_id)
```

#### 8.3.3 检索未完成时的处理

当用户连续快速提问：

```
用户: "RAG 的原理是什么？"     → 触发异步检索
用户: "召回阈值怎么设置？"     ← 检索尚未完成
```

**处理策略**：直接使用当前 memory_context（可能为空），不阻塞响应。

下次对话时，memory_context 已更新，自动使用新记忆。

### 8.4 记忆注入流程

```python
async def build_context_with_memory(state: AgentState) -> list[BaseMessage]:
    """构建完整上下文（包含记忆）
    
    在每次对话开始前调用。
    """
    
    messages = state.get("messages", [])
    memory_context = state.get("memory_context", "")
    
    if memory_context:
        memory_message = SystemMessage(
            content=f"<记忆上下文>\n{memory_context}\n</记忆上下文>"
        )
        messages.append(memory_message)
    
    return messages
```

**注入时机**：Agent 节点开始处理用户问题前。

### 8.5 记忆写入流程

#### 8.5.1 session_summaries 写入

```python
async def save_session_summary(conversation_id: str, user_id: str):
    """对话结束时，写入 session_summaries"""
    
    messages = pg.get_messages(user_id, conversation_id)
    
    # 1. LLM 生成摘要
    summary = await generate_summary(messages)
    
    # 2. 计算 embedding
    embedding = get_embedding_tool().embed_text(summary)
    
    # 3. 判断类型
    session_type = detect_session_type(messages)
    
    # 4. 获取 metadata
    metadata = get_interview_metadata(conversation_id) if session_type == "interview" else {}
    
    # 5. 写入
    pg.create_session_summary(
        conversation_id, user_id, summary, embedding, session_type, metadata
    )
    
    # 6. 同步反馈到 preferences.md
    feedback = extract_feedback_from_summary(summary)
    if feedback:
        await update_preferences_from_feedback(user_id, feedback)
    
    # 7. 更新 behaviors.md
    await update_behaviors_from_conversation(user_id, conversation_id)
    
    # 8. 同步 MEMORY.md 概要
    await update_memory_index(user_id)
```

#### 8.5.2 preferences.md 更新

触发时机：用户表达反馈时或对话结束时

```python
async def update_preferences_from_feedback(user_id: str, feedback: str):
    """从用户反馈更新 preferences.md"""
    
    current = read_memory_reference(user_id, "preferences")
    
    prompt = f"""
当前用户偏好文档：
{current}

用户新反馈：
"{feedback}"

请更新用户偏好文档：
1. 将新反馈整合到对应部分
2. 在"反馈历史"部分添加记录
"""
    
    updated = await llm.ainvoke(prompt)
    write_memory_reference(user_id, "preferences", updated)
    
    await update_memory_index(user_id)
```

#### 8.5.3 behaviors.md 更新

触发时机：对话结束时

```python
async def update_behaviors_from_conversation(user_id: str, conversation_id: str):
    """从对话分析用户行为"""
    
    messages = pg.get_messages(user_id, conversation_id)
    current = read_memory_reference(user_id, "behaviors")
    
    prompt = f"""
当前用户行为模式：
{current}

本次对话：
{format_messages(messages)}

请分析并更新行为模式文档。
"""
    
    updated = await llm.ainvoke(prompt)
    write_memory_reference(user_id, "behaviors", updated)
```

#### 8.5.4 MEMORY.md 概要同步

```python
async def update_memory_index(user_id: str):
    """同步更新 MEMORY.md 概要"""
    
    preferences = extract_summary(read_memory_reference(user_id, "preferences"))
    behaviors = extract_summary(read_memory_reference(user_id, "behaviors"))
    recent_sessions = pg.get_recent_session_summaries(user_id, limit=5)
    
    prompt = f"""
请更新 MEMORY.md 的概要部分：
- 偏好概要：{preferences}
- 行为模式概要：{behaviors}
- 会话历史概要：{format_session_overview(recent_sessions)}

确保文档控制在 300-500 tokens。
"""
```

## 十、并发控制

### 10.1 检索锁（Redis）

**锁粒度**：会话级别（conversation_id）

**目的**：同一个会话内不重复触发检索

```python
def get_memory_retrieval_lock_key(user_id: str, conversation_id: str) -> str:
    return f"memory_retrieval:{user_id}:{conversation_id}"

async def acquire_memory_retrieval_lock(user_id: str, conversation_id: str) -> bool:
    lock_key = get_memory_retrieval_lock_key(user_id, conversation_id)
    acquired = redis_client.setnx(lock_key, "locked")
    if acquired:
        redis_client.expire(lock_key, settings.memory_retrieval_lock_timeout)
    return acquired
```

### 10.2 更新锁（Redis）

**锁粒度**：用户级别 + 类型（user_id + lock_type）

**目的**：防止多个 Worker 同时更新 preferences.md / behaviors.md

```python
async def acquire_memory_lock(user_id: str, lock_type: str) -> bool:
    lock_key = f"memory_lock:{user_id}:{lock_type}"
    acquired = redis_client.setnx(lock_key, "locked")
    if acquired:
        redis_client.expire(lock_key, MEMORY_LOCK_TIMEOUT)
    return acquired

async def update_with_lock(user_id: str, lock_type: str, update_func: Callable):
    """带锁的更新操作（重试 3 次）"""
    for attempt in range(3):
        if await acquire_memory_lock(user_id, lock_type):
            try:
                await update_func()
            finally:
                await release_memory_lock(user_id, lock_type)
            return
        await asyncio.sleep(5)
```

## 十一、错误处理与降级

### 11.1 重试策略

- 最大重试次数：3 次
- 重试间隔：递增（1s → 3s → 5s）

### 11.2 降级策略

| 错误类型 | 处理方式 | 影响 |
|---------|---------|---------|
| LLM 调用失败 | 重试 → 降级使用 title | 仅影响 summary 质量 |
| Embedding 失败 | 重试 → 写入 NULL | 影响语义检索 |
| 数据库写入失败 | 重试 → 放弃本次写入 | 丢失本次摘要，不影响对话 |
| 锁获取失败 | 重试 → 放弃本次更新 | preferences/behaviors 暂不更新 |

### 11.3 降级示例

```python
async def save_session_summary(conversation_id: str, user_id: str):
    messages = pg.get_messages(user_id, conversation_id)
    conv = pg.get_conversation(user_id, conversation_id)
    
    # 生成 summary（重试 + 降级）
    success, summary = await retry_with_backoff(generate_summary, messages)
    if not success:
        summary = f"主题：{conv.title}"  # 降级使用 title
    
    # 计算 embedding（重试 + 降级）
    success, embedding = await retry_with_backoff(embed_text, summary)
    if not success:
        embedding = None  # 允许 NULL
    
    # 写入（重试 + 放弃）
    success, error = await retry_with_backoff(pg.create_session_summary, ...)
    if not success:
        return  # 放弃，不影响用户
```

## 十八、核心设计决策总结

| 决策 | 原因 |
|------|------|
| **记忆 = Skill** | 遵循行业标准，概念统一 |
| **MEMORY.md + references** | 渐进式披露，节省 token |
| **session_summaries 独立表** | 数据 vs 规则分离，支持检索 |
| **CASCADE DELETE** | 删除对话时自动清理记忆 |
| **Worker 直接更新 Checkpointer** | 状态统一管理，无需中间缓存 |
| **检索异步执行** | 不阻塞主流程，记忆是增强功能 |
| **preferences.md 管理偏好** | 避免 UserSettings 同步冲突 |
| **Redis 锁控制并发** | 防止检索/更新冲突 |
