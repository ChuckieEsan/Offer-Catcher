## 一、设计理念

### 1.1 核心洞察

> 记忆本质上就是用户自定义的 Skill——指导 Agent 行为的用户特定规则文档。

因此，记忆模块完全遵循 [Agent Skills 标准](https://agentskills.io/specification)：
- 主文档（MEMORY.md）+ references 目录的结构
- 渐进式披露（Progressive Disclosure）加载机制
- LLM 直接阅读、编辑、按需查询

### 1.2 与全局 Skill 的对比

| 方面 | 全局 Skill | 用户记忆（MEMORY） |
|------|-----------|-------------------|
| **范围** | 所有用户共享 | 单用户特定 |
| **位置** | `.agents/skills/` | `memory/{user_id}/` |
| **结构** | SKILL.md + references | MEMORY.md + references |
| **加载** | 按触发条件加载 | MEMORY.md 始终加载，references 按需查询 |
| **创建方式** | 开发者编写 | 用户反馈积累 / UI 创建 |

## 二、数据边界划分

### 2.1 存储分类

| 数据 | 存储位置 | 内容 | 用途 |
|------|---------|------|------|
| **mastery_level** | Qdrant | 题目掌握程度（LEVEL_0/1/2） | 客观事实，判断薄弱题目 |
| **preferences.md** | 记忆 references | 用户显式表达的偏好 | 指导响应风格 |
| **behaviors.md** | 记忆 references | 用户交互行为模式 | 指导响应策略 |
| **session_summaries** | 数据库表 | 会话摘要 + 向量 | 支持历史回溯、语义检索 |

### 2.2 核心原则

- **规则**（preferences/behaviors）→ Markdown 文件，放入 references，LLM 按需调用
- **数据**（session_summaries）→ 数据库表，支持向量检索
- **消息**（messages）→ 数据库表，完整历史，用于生成摘要

```

│

│   # - 行为模式概要

│   # - 可用 references 列表
│   # - 可用 Skill 列表

    ├── preferences.md            # 用户偏好规则

    │   # - 话题偏好
    │   # - 反馈历史
    │
    ├── behaviors.md              # 行为模式规则
    │   # - 提问序列 + 建议
```

必须遵循 Agent Skills 标准：


## 四、与现有模块的关系

### 4.1 继续保留的模块

| 模块 | 用途 | 与记忆的关系 |
|------|------|-------------|
| conversations | 对话元信息 | session_summaries 通过 conversation_id 关联 |
| messages | 完整消息内容 | 用于生成 summary、embedding |
| UserProfile | 用户身份信息 | 独立存在，不与记忆同步 |
| Qdrant mastery_level | 题目掌握程度 | 独立存在，不放入记忆 |

### 4.2 新增模块

| 模块 | 位置 | 用途 |
|------|------|------|
| MEMORY.md | PostgresStore | 主文档（始终加载） |
| preferences.md | PostgresStore | 用户偏好规则 |
| behaviors.md | PostgresStore | 行为模式规则 |
| skills/{skill}/ | PostgresStore | 用户自定义 Skill |
| session_summaries | 数据库表 | 会话摘要 + 向量 |

### 4.3 preferences.md 的职责定位

**核心决策**：Agent 响应偏好完全由 `preferences.md` 管理，用户通过记忆页面直接编辑。

**设计原因**：
- 避免数据同步冲突（不引入额外的 UserSettings 系统）
- 用户直接编辑 preferences.md 更直观（所见即所得）
- Agent 直接读取 preferences.md，无需通过中间层中转

### 4.4 数据流向图

```
用户对话
    ↓
messages 表（存储完整消息）
    ↓
对话结束 → 异步 Worker
    ↓
LLM 生成 → summary + embedding
    ↓
session_summaries 表（CASCADE DELETE）
    ↓
从 summary 提取反馈 → preferences.md
从对话分析 → behaviors.md
    ↓
MEMORY.md（同步更新概要）
```

## 五、MEMORY.md 主文档设计

### 5.1 格式规范

```markdown
---
name: user-memory-{user_id}
description: 用户特定的偏好和行为规则。始终加载此文档。
             当概要信息不足以回答问题时，使用 load_memory_reference Tool
             或 search_session_history Tool 查询详情。
---

# 用户记忆

## 偏好概要
- 解释深度：深入详细
- 代码示例：偏好给出代码示例
- 语言：中文

## 行为模式概要
- 提问序列：问 what 后常问 why
- 关注焦点：偏好实践应用
- 追问风格：追问具体细节

## 会话历史概要
最近 5 次会话：
- 2026-04-12 | 面试模拟复盘（字节跳动 Agent 岗，得分 65）
- 2026-04-11 | Agent 架构对比
- 2026-04-10 | RAG 原理讨论

需要更多历史详情时，使用 `search_session_history` Tool。

## 可用 References
| Reference | 描述 | 建议调用时机 |
|-----------|------|-------------|
| `preferences` | 完整的用户偏好设置 | 用户表达反馈 / 回答需要个性化调整 |
| `behaviors` | 观察到的行为模式详情 | Agent 需要调整响应策略 |

## 可用自定义 Skill
| Skill | 描述 | 触发条件 |
|-------|------|---------|
| `interview_tips` | 面试技巧 | 模拟面试开始时 |

## 使用指南
1. 本文档始终加载，提供概要信息
2. 概要不够详细时，调用 `load_memory_reference("{reference_name}")`
3. 需要语义检索历史时，调用 `search_session_history("{query}")`
4. 触发自定义 Skill 时，调用 `load_skill("{skill_name}")`
```

### 5.2 长度控制

目标：**300-500 tokens**

- 偏好概要：~50 tokens（3-5 条要点）
- 行为模式概要：~100 tokens（3-4 条要点）
- 会话历史概要：~100 tokens（最近 5 条）
- References/Skill 列表：~150 tokens

## 六、references 详情设计

### 6.1 preferences.md

```markdown
# 用户偏好详情

## 响应风格
- 语言：中文
- 解释深度：深入详细，尤其针对生疏知识点要鞭辟入里
- 响应长度：适中，但允许长文解释复杂概念
- 代码示例：偏好给出代码示例

## 话题偏好

### RAG
- 必须给出代码示例
- 优先解释实践细节（召回阈值、重排序），而非向量原理
- 涉及召回阈值时，给出具体数值范围（如 0.7-0.85）

### Agent
- 优先实现细节和错误处理
- 对比 LangChain vs LangGraph 时，给出表格形式

### Python
- 跳过基础语法解释
- 重点讲解架构设计和最佳实践

## 反馈历史
| 日期 | 反馈内容 | 影响 |
|------|---------|------|
| 2026-04-10 | "RAG 解释不够深入，缺少代码" | 添加 RAG 话题偏好：必须代码示例 |
| 2026-04-12 | "解释鞭辟入里" | 更新响应风格：解释深度为深入详细 |
```

### 6.2 behaviors.md

```markdown
# 用户行为模式详情

## 提问序列
用户经常在问"什么是 X"之后追问"为什么需要 X"或"X 怎么实现"。

**建议**：回答 what 类问题时，主动补充 why 和 how 的信息。

## 关注焦点
用户偏好 **practical_use**（实践应用）类型的答案。

**证据**：用户追问"怎么用"、"代码怎么写"的频率高于"原理是什么"。

**建议**：检索时优先返回带代码示例的题目。

## 追问风格
用户的追问通常针对 **具体细节**（参数、边界情况、错误处理）。

**建议**：回答时主动覆盖常见边界情况、错误处理方式。

## 知识背景推测
用户有一定 Python 基础，对 LangChain API 有使用经验，但对架构设计理解不深。

**建议**：可以跳过 Python 基础解释，但架构层面要详细展开。
```

## 七、session_summaries 表设计

### 7.1 设计理念

> session_summaries 存储对话中产生的记忆摘要，用于语义检索历史。
>
> **一个 conversation 可以有多条 summary**：每轮有价值的对话都可能产生一条记忆。

| 表 | 职责 | 存储内容 |
|---|------|---------|
| **conversations** | 对话管理 | title, created_at, updated_at |
| **messages** | 消息存储 | 完整的对话消息内容 |
| **session_summaries** | 记忆检索 | summary, embedding（一个 conversation 可有多条） |

### 7.2 表结构

```sql
CREATE TABLE session_summaries (
    id VARCHAR(36) PRIMARY KEY,                    -- 主键（UUID）
    conversation_id VARCHAR(36) NOT NULL,          -- 关联对话（外键）
    user_id VARCHAR(36) NOT NULL,
    summary TEXT NOT NULL,                         -- 3句话摘要
    embedding vector(1024),                        -- 语义向量
    message_cursor_uuid VARCHAR(36),               -- 产生此记忆时的游标位置
    created_at TIMESTAMP NOT NULL DEFAULT NOW(),
    FOREIGN KEY (conversation_id) REFERENCES conversations(id) ON DELETE CASCADE
);

-- 索引
CREATE INDEX idx_session_summaries_user_id ON session_summaries(user_id);
CREATE INDEX idx_session_summaries_conversation_id ON session_summaries(conversation_id);
CREATE INDEX idx_session_summaries_user_created ON session_summaries(user_id, created_at DESC);

-- 向量索引
CREATE INDEX idx_session_summaries_embedding ON session_summaries 
USING ivfflat (embedding vector_cosine_ops) WITH (lists = 100);
```

### 7.3 字段说明

| 字段 | 类型 | 说明 |
|------|------|------|
| id | VARCHAR(36) | 主键（UUID） |
| conversation_id | VARCHAR(36) | 关联 conversations（一个 conversation 可有多条） |
| user_id | VARCHAR(36) | 用户 ID |
| summary | TEXT | 会话摘要（关键问题和讨论要点，3句话左右） |
| embedding | vector(1024) | 语义向量（用于检索） |
| message_cursor_uuid | VARCHAR(36) | 产生此记忆时处理的游标位置（用于追踪） |
| created_at | TIMESTAMP | 创建时间 |

### 7.4 一对多关系

一个 conversation 的多轮对话示例：

```
Conversation A:
  ├─ Round 1: 用户问 RAG → 记忆 1 (summary: "讨论 RAG 原理...")
  ├─ Round 2: 用户追问阈值 → 记忆 2 (summary: "讨论召回阈值...")
  └─ Round 3: 用户说"记住我喜欢具体数值" → 记忆 3 (summary: "用户偏好具体数值...")
```

### 7.5 CASCADE DELETE

删除 conversation 时自动删除所有关联的 session_summaries，保证数据一致性。

**注意**：preferences.md 和 behaviors.md **不会被删除**，因为有价值的内容已被吸收进去。

## 八、记忆生命周期

### 8.1 整体流程图

```
┌─────────────────────────────────────────────────────────────┐
│ 用户注册 / 首次对话                                          │
│                                                              │
│ 初始化：创建 MEMORY.md + preferences.md + behaviors.md        │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│ 对话结束 → Stop Hook 触发                                     │
│                                                              │
│ 启动记忆 Agent：                                              │
│ 传入：messages + 游标（last_memory_message_uuid）            │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│ 记忆 Agent：只处理游标之后的新消息                             │
│                                                              │
│ ├─ 有记忆内容 →                                              │
│ │   ├─ summary → session_summaries 表 + embedding            │
│ │   ├─ preferences_content → preferences.md（完整内容）       │
│ │   ├─ behaviors_content → behaviors.md（完整内容）           │
│ │   └─ 更新 MEMORY.md 概要                                   │
│ │   └─ 更新游标到最新消息                                     │
│ │                                                            │
│ └─ 无记忆内容 → 跳过，不处理                                  │
└─────────────────────────────────────────────────────────────┘
                              ↓
                      下一轮对话（重复流程）
```

### 8.2 游标机制

**游标定义**：`last_memory_message_uuid` 是一个消息 UUID，记录上次处理的位置。

```
┌────────────────────────────────────────────────────────────┐
│  已处理区域（游标之前）  │  待处理区域（游标之后）          │
│  last_memory_message_uuid  │  新消息，需要分析是否有记忆内容│
└────────────────────────────────────────────────────────────┘
```

**作用**：
- 划定处理边界，避免重复处理
- 与游标互斥机制配合（主 Agent 写入后跳过）

### 8.3 记忆 Agent 输出结构

记忆 Agent 分析游标后的新消息，输出 JSON：

```json
{
  "skip": false,
  "session_summary": {
    "summary": "用户询问了 RAG 的召回阈值设置，讨论了 0.7-0.85 的推荐范围。"
  },
  "preferences_content": "# 用户偏好详情\n\n## 响应风格\n...",
  "behaviors_content": "# 用户行为模式详情\n\n## 提问序列\n..."
}
```

| 字段 | 说明 |
|------|------|
| skip | 是否跳过（无记忆内容时为 true） |
| session_summary.summary | 3句话摘要，写入 session_summaries 表 |
| preferences_content | 完整的 preferences.md 内容（如有更新） |
| behaviors_content | 完整的 behaviors.md 内容（如有更新） |

### 8.4 讝始化流程

**触发时机**：
- 用户注册时（自动）
- 首次对话时检测到记忆不存在（兜底）

**实现**：

```python
async def initialize_user_memory(user_id: str) -> None:
    """初始化用户记忆"""
    
    if read_memory(user_id):
        return
    
    # 创建 MEMORY.md
    memory_content = MEMORY_TEMPLATE.replace("{user_id}", user_id)
    write_memory(user_id, memory_content)
    
    # 创建 preferences.md
    write_memory_reference(user_id, "preferences", PREFERENCES_TEMPLATE)
    
    # 创建 behaviors.md
    write_memory_reference(user_id, "behaviors", BEHAVIORS_TEMPLATE)
```

**默认模板**（见附录）。

### 8.5 记忆 Agent Prompt 设计

**文件位置**：`backend/app/agents/prompts/memory_agent.md`

Memory Agent 通过 `create_agent` 创建，拥有自己的 tools，自主完成记忆修改。Prompt 使用 `<tools>` 标签详细描述每个工具的功能和参数。

```markdown
---
name: memory-agent
description: 从对话中提取有价值信息并更新用户记忆。在对话结束时自动触发。
---

<role>
你是一个记忆管理 Agent，负责从对话中提取有价值的信息并更新用户记忆。
</role>

<task>
分析游标之后的新消息，提取记忆内容并调用相应的工具完成更新。
</task>

<tools>
<!-- write_session_summary: WHEN 有值得记录的会话内容 -->
<tool name="write_session_summary">
写入会话摘要到数据库（用于语义检索历史）
参数：
- summary: 会话摘要（简洁描述关键内容，如"用户询问了 RAG 的召回阈值设置"）
- conversation_id: 对话 ID
- user_id: 用户 ID
</tool>

<!-- update_preferences: WHEN 用户表达了偏好或反馈 -->
<tool name="update_preferences">
更新用户偏好设置文件
参数：
- content: 完整的 preferences.md 内容（整合现有内容和新反馈）
- user_id: 用户 ID
</tool>

<!-- update_behaviors: WHEN 观察到用户的行为模式 -->
<tool name="update_behaviors">
更新用户行为模式文件
参数：
- content: 完整的 behaviors.md 内容（整合现有内容和新观察）
- user_id: 用户 ID
</tool>

<!-- update_memory_index: WHEN preferences 或 behaviors 有更新时 -->
<tool name="update_memory_index">
更新 MEMORY.md 概要（同步最新偏好和行为概要）
参数：
- user_id: 用户 ID
</tool>

<!-- update_cursor: ALWAYS 最后必须调用 -->
<tool name="update_cursor">
更新游标位置（标记已处理到最新消息）
参数：
- conversation_id: 对话 ID
- user_id: 用户 ID
- cursor_uuid: 最新消息的 UUID
</tool>
</tools>

<rules>
- 分析消息后，根据内容调用相应工具
- 如果没有值得记忆的内容，直接调用 update_cursor 结束
- 如果有记忆内容，先调用需要的工具，最后调用 update_cursor
- preferences/behaviors 传入完整的 md 文件内容（而非追加）
</rules>

<context>
游标后的新消息：
{{ new_messages }}

当前 preferences.md：
{{ current_preferences }}

当前 behaviors.md：
{{ current_behaviors }}

conversation_id: {{ conversation_id }}
user_id: {{ user_id }}
cursor_uuid: {{ cursor_uuid }}
</context>
```

**关键设计要点**：
- **`<tools>` 标签详细描述**：每个工具的 WHEN 条件、参数、用途
- **不要求输出格式**：Prompt 描述任务和工具，Agent 自主调用
- **context 包含上下文**：游标后的消息、当前 preferences/behaviors、conversation_id 等
- **Agent 通过 ReAct 模式**：自主分析 → 调用工具 → 更新游标

### 8.6 记忆 Agent 实现

**文件位置**：`backend/app/memory/agent.py`

使用 LangChain 的 `create_agent` 创建 Agent：

```python
from langchain.agents import create_agent
from langchain.messages import SystemMessage

def create_memory_agent():

async def run_memory_agent(
    user_id: str,
    conversation_id: str,
    messages: list[BaseMessage],
    cursor: str | None,
):
    """执行记忆 Agent
    
    Agent 自主分析消息并调用 tools 完成记忆更新。
    """
    # 1. 检查游标互斥
    if has_memory_writes_since(messages, cursor):
        logger.info("Memory update skipped: main agent already wrote")
        return
    
    # 2. 获取游标后的新消息
    new_messages = get_messages_since_cursor(messages, cursor)
    
    if not new_messages:
        return
    
    # 3. 获取当前内容
    current_preferences = read_memory_reference(user_id, "preferences")
    current_behaviors = read_memory_reference(user_id, "behaviors")
    
    # 4. 构建 Prompt 参数
    prompt_kwargs = {
        "new_messages": format_messages(new_messages),
        "current_preferences": current_preferences,
        "current_behaviors": current_behaviors,
    }
    
    # 5. 创建带有上下文的 Agent
    agent = create_memory_agent()
    
    # 6. 构建输入消息（包含上下文）
    input_messages = [
        HumanMessage(content="请分析以下对话并更新记忆。"
                    f"\n\nconversation_id: {conversation_id}"
                    f"\nuser_id: {user_id}"
                    f"\n游标位置: {cursor or '初始'}"),
    ]
    
    # 7. 调用 Agent（Agent 自主调用 tools）
    result = await agent.ainvoke({"messages": input_messages})
    
    # Agent 通过 tools 自主完成了所有记忆更新
```

### 8.7 记忆检索流程

#### 8.7.1 触发条件

- 消息长度 ≥ 10 个字符
- 非空字符串
- 每次有效提问触发检索

#### 8.7.2 异步检索 Worker

```python
async def retrieve_memory_worker(
    user_id: str,
    conversation_id: str,
    query: str,
    checkpointer: PostgresSaver,
):
    """异步检索记忆
    
    流程：
    1. 计算 query embedding
    2. 语义检索 session_summaries（top_k = 5）
    3. 获取对话标题
    4. 格式化为 memory_context
    5. 读取当前 checkpoint，更新 memory_context
    """
    settings = get_settings()
    
    try:
        # 1. 计算 embedding
        query_embedding = get_embedding_tool().embed_text(query)
        
        # 2. 语义检索
        results = pg.search_session_summaries(
            user_id=user_id,
            query_embedding=query_embedding,
            top_k=settings.memory_retrieval_top_k,
        )
        
        # 3. 获取标题
        for r in results:
            conv = pg.get_conversation(user_id, r["conversation_id"])
            if conv:
                r["title"] = conv.title
                r["created_at"] = conv.created_at
        
        # 4. 更新 checkpoint
        config = {"configurable": {"thread_id": conversation_id}}
        current_checkpoint = await checkpointer.aget(config)
        
        if current_checkpoint:
            current_memory = current_checkpoint.channel_values.get("memory_context", "")
            injected_ids = current_checkpoint.channel_values.get("injected_session_ids", [])
            
            merged_context, merged_ids = merge_memory_context(
                current_memory, injected_ids, results
            )
            
            current_checkpoint.channel_values["memory_context"] = merged_context
            current_checkpoint.channel_values["injected_session_ids"] = merged_ids
            
            await checkpointer.aput(config, current_checkpoint)
        
    finally:
        await release_memory_retrieval_lock(user_id, conversation_id)
```

#### 8.7.3 检索未完成时的处理

当用户连续快速提问：

```
用户: "RAG 的原理是什么？"     → 触发异步检索
用户: "召回阈值怎么设置？"     ← 检索尚未完成
```

**处理策略**：直接使用当前 memory_context（可能为空），不阻塞响应。

下次对话时，memory_context 已更新，自动使用新记忆。

### 10.1 检索锁（Redis）

```python
    return f"memory_retrieval:{user_id}:{conversation_id}"

async def acquire_memory_retrieval_lock(user_id: str, conversation_id: str) -> bool:
    lock_key = get_memory_retrieval_lock_key(user_id, conversation_id)
    acquired = redis_client.setnx(lock_key, "locked")
    if acquired:
        redis_client.expire(lock_key, settings.memory_retrieval_lock_timeout)
```

### 10.2 更新锁（Redis）

```python
async def update_with_lock(user_id: str, lock_type: str, update_func: Callable):
    """带锁的更新操作（重试 3 次）"""
    for attempt in range(3):
        if await acquire_memory_lock(user_id, lock_type):
            try:
                await update_func()
            finally:
                await release_memory_lock(user_id, lock_type)
            return
        await asyncio.sleep(5)
```

## 十一、错误处理与降级

### 11.1 重试策略

- 最大重试次数：3 次
- 重试间隔：递增（1s → 3s → 5s）

### 11.2 降级策略

| 错误类型 | 处理方式 | 影响 |
|---------|---------|---------|
| LLM 调用失败 | 重试 → 降级使用 title | 仅影响 summary 质量 |
| Embedding 失败 | 重试 → 写入 NULL | 影响语义检索 |
| 数据库写入失败 | 重试 → 放弃本次写入 | 丢失本次摘要，不影响对话 |
| 锁获取失败 | 重试 → 放弃本次更新 | preferences/behaviors 暂不更新 |

### 11.3 降级示例


## 十二、容量管理

### 12.1 memory_context 容量控制

- 上限：20kb
- 超过时裁剪最早注入的记忆
- 使用 conversation_id 去重

```python
def merge_memory_context(current_memory, injected_ids, new_results):
    max_size = settings.memory_context_max_size
    
    # 去重 + 合并
    ...
    
    # 容量控制
    if len(merged_memory.encode('utf-8')) > max_size:
        sessions = merged_memory.split("\n\n### ")
        while len(merged_memory) > max_size and len(sessions) > 1:
            sessions.pop(0)
            injected_ids.pop(0)
        merged_memory = "\n\n### ".join(sessions)
    
    return merged_memory, injected_ids
```

### 12.2 文件压缩规则

| 文件 | 压缩规则 | 目标长度 |
|------|---------|---------|
| MEMORY.md | LLM 更新时自动精简 | ~300-500 tokens |
| preferences.md | 反馈历史保留最近 20 条 | ~800-1000 tokens |
| behaviors.md | 只保留高置信度模式 | ~1000-1500 tokens |

### 12.3 session_summaries 表

**不压缩**：每条记录独立存储，支持语义检索和 CASCADE DELETE。

### 12.4 每次注入增量控制

**设计目标**：控制每次检索注入的记忆增量，避免上下文膨胀。

| 参数 | 值 | 说明 |
|------|---|------|
| `memory_retrieval_top_k` | 5 | 每次检索最多返回 5 条会话记忆 |
| 单条摘要长度 | ~50 字（~100 tokens） | 简洁描述关键内容 |
| 单次注入增量上限 | **~500 tokens** | 5 条 × 100 tokens |
| 去重机制 | conversation_id | 同一对话的摘要不重复注入 |

**去重逻辑**：

```python
def merge_memory_context(
    current_memory: str,
    injected_ids: list[str],  # 已注入的 conversation_id
    new_results: list[dict],  # 新检索结果
) -> tuple[str, list[str]]:
    """合并记忆上下文（去重 + 容量控制）"""
    
    # 去重：跳过已注入的 conversation_id
    for r in new_results:
        if r["conversation_id"] not in injected_ids:
            injected_ids.append(r["conversation_id"])
            current_memory += format_session_summary(r)
    
    # 容量控制：超过上限时裁剪最早的
    if len(current_memory) > settings.memory_context_max_size:
        # 按注入顺序裁剪
        while len(current_memory) > settings.memory_context_max_size:
            injected_ids.pop(0)  # 移除最早的
            current_memory = rebuild_memory_from_ids(injected_ids)
    
    return current_memory, injected_ids
```

**关键要点**：
- 每次检索最多注入 5 条新记忆
- 通过 conversation_id 去重，同一对话的摘要不会重复添加
- 单次增量控制在约 500 tokens（5 条 × 100 tokens）
- 累积上限 20kb，超过时裁剪最早注入的记忆

## 十八、核心设计决策总结

| 决策 | 原因 |
|------|------|
| **记忆 = Skill** | 遵循行业标准，概念统一 |

| **session_summaries 独立表** | 数据 vs 规则分离，支持检索 |

| **CASCADE DELETE** | 删除对话时自动清理 session_summaries |
| **create_react_agent 创建 Agent** | Agent 拥有自己的 tools，自主完成记忆修改 |
| **Tools 执行操作** | 每个 tool 负责实际写入/更新，Agent 调用 tools |
| **检索异步执行** | 不阻塞主流程，记忆是增强功能 |
| **preferences.md 管理偏好** | 避免 UserSettings 同步冲突，preferences/behaviors 保留 |
| **Redis 锁控制并发** | 防止检索/更新冲突 |
| **游标机制** | 避免重复处理，划定处理边界 |
| **游标互斥** | 保证主 Agent 与后台 Agent 写入不冲突 |

## 十九、游标互斥机制

> 本机制保证主 Agent 和后台 Agent 写入记忆的并发安全性。

### 19.1 问题背景

主 Agent 和后台 Agent 是完全独立的异步进程：

```
第 N 轮：
用户输入 → 主 Agent 响应 → Stop Hook
                ↓
        void extractMemories() ← fire-and-forget
                ↓
        后台 Agent N 开始执行（可能耗时几秒）

第 N+1 轮（后台 Agent N 还在执行）：
用户: "请记住我喜欢用 bun"
                ↓
        主 Agent 立即执行 Write（不等待后台 Agent）
                ↓
        写入完成 → Stop Hook
```

**潜在问题**：
- 主 Agent 和后台 Agent 可能同时写入 preferences.md
- 后台 Agent 的更新可能覆盖主 Agent 的写入

### 19.2 游标机制

**游标定义**：`last_memory_message_uuid` 是一个消息 UUID，记录上次处理的位置。

```
┌────────────────────────────────────────────────────────────┐
│  已处理区域（游标之前）  │  待处理区域（游标之后）          │
│  last_memory_message_uuid  │  新消息，需要分析是否提取记忆  │
└────────────────────────────────────────────────────────────┘
```

### 19.3 互斥检查

```python
def has_memory_writes_since(
    messages: list[BaseMessage],
    since_uuid: str,
) -> bool:
    """检查游标之后是否有主 Agent 记忆写入
    
    Args:
        messages: 消息列表
        since_uuid: 上次处理的消息 UUID
        
    Returns:
        是否有记忆写入
    """
    found_start = False
    
    for msg in messages:
        if not found_start:
            if hasattr(msg, 'uuid') and msg.uuid == since_uuid:
                found_start = True
            continue
        
        # 只检查游标之后的 assistant 消息
        if getattr(msg, 'type', '') == 'assistant':
            content = getattr(msg, 'content', '')
            # 检查是否有记忆写入标记
            if '<memory_write>' in content:
                return True
    
    return False
```

### 19.4 互斥流程

```python
async def update_memory_with_mutex(
    user_id: str,
    conversation_id: str,
    messages: list[BaseMessage],
    last_uuid: str,
):
    """带互斥的记忆更新
    
    流程：
    1. 检查游标之后是否有主 Agent 写入
    2. 如果有，跳过后台更新
    3. 否则，执行后台更新
    4. 更新游标到最新消息
    """
    if has_memory_writes_since(messages, last_uuid):
        logger.info("Memory update skipped: main agent already wrote")
        return
    
    # 执行后台更新
    await _execute_memory_update(user_id, conversation_id, messages)
    
    # 更新游标
    new_uuid = messages[-1].uuid if messages else last_uuid
    await update_memory_cursor(user_id, conversation_id, new_uuid)
```

### 19.5 主 Agent 写入标记

主 Agent 直接写入记忆时，在响应中添加标记：

```python
# 主 Agent 写入时
write_memory_reference(user_id, "preferences", updated_content)

# 在响应中添加标记（后台 Agent 检查此标记）
response = f"<memory_write>preferences</memory_write>\n已更新您的偏好设置。"
```

### 19.6 并发场景分析

| 问题 | 答案 |
|------|------|
| 主 Agent 会等待后台 Agent 吗？ | ❌ **不会**，完全异步 |
| 主 Agent 写入会被阻塞吗？ | ❌ **不会**，独立执行 |
| 会有冲突吗？ | ❌ **不会**，游标互斥保证不重复 |
| 会丢失记忆吗？ | ❌ **不会**，主 Agent 已写入用户请求的记忆 |

### 19.7 核心要点

- **游标互斥**：基于消息 UUID 的协议，而非锁或等待
- **主 Agent 优先**：主 Agent 写入后，后台 Agent 检查到就跳过
- **写入标记**：主 Agent 在响应中添加 `<memory_write>` 标记
- **游标推进**：处理完成后更新游标到最新消息
