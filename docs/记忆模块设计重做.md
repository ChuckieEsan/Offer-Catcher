# Offer-Catcher 记忆模块设计（最终版）

> 本文档基于 2026-04-13 的讨论，最终确定记忆模块架构。核心概念：**记忆 = 用户自定义 Skill**，遵循 Skill 标准。

---

## 一、核心设计理念

### 1.1 记忆即 Skill

**核心洞察**：
> 记忆本质上就是用户自定义的 Skill——指导 Agent 行为的用户特定规则文档。

因此，记忆模块完全遵循 [Agent Skills 标准](https://agentskills.io/specification)：
- 主文档（MEMORY.md）+ references 目录的结构
- 渐进式披露（Progressive Disclosure）加载机制
- LLM 直接阅读、编辑、按需查询

### 1.2 与 OpenAI/Claude Skill 的关系

| 方面 | 全局 Skill | 用户记忆（MEMORY） |
|------|-----------|-------------------|
| **范围** | 所有用户共享 | 单用户特定 |
| **位置** | `.agents/skills/` | `memory/{user_id}/` |
| **结构** | SKILL.md + references | MEMORY.md + references |
| **加载** | 按触发条件加载 | MEMORY.md 始终加载，references 按需查询 |
| **创建方式** | 开发者编写 | 用户反馈积累 / UI 创建 |

### 1.3 数据边界划分

| 数据 | 存储位置 | 内容 | 用途 |
|------|---------|------|------|
| **mastery_level** | Qdrant | 题目掌握程度（LEVEL_0/1/2） | 客观事实，判断薄弱题目 |
| **preferences.md** | 记忆 references | 用户显式表达的偏好 | 指导响应风格 |
| **behaviors.md** | 记忆 references | 用户交互行为模式 | 指导响应策略 |
| **session_summaries** | 数据库表 | 会话摘要 + 向量 | 支持历史回溯、语义检索 |

**核心原则**：
- **规则**（preferences/behaviors）→ Markdown 文件，放入 references
- **数据**（session_summaries）→ 数据库表，支持检索

## 三、MEMORY.md 主文档设计

### 3.1 格式规范

MEMORY.md 遵循 Skill 标准的 frontmatter + 内容结构：

```markdown
---
name: user-memory-{user_id}
description: 用户特定的偏好和行为规则。始终加载此文档。
             当概要信息不足以回答问题时，使用 load_memory_reference Tool
             或 search_session_history Tool 查询详情。
---

# 用户记忆

## 偏好概要
- 解释深度：深入详细（偏好鞭辟入里）
- 代码示例：偏好给出代码示例
- 语言：中文

## 行为模式概要
- 提问序列：问 what 后常问 why
- 关注焦点：偏好实践应用
- 追问风格：追问具体细节（参数、边界情况）

## 会话历史概要
最近 5 次会话：
- 2026-04-12 | 面试模拟复盘（字节跳动 Agent 岗，得分 65）
- 2026-04-11 | Agent 架构对比
- 2026-04-10 | RAG 原理讨论
- 2026-04-08 | Redis 缓存策略
- 2026-04-05 | LangChain 入门

需要更多历史详情时，使用 `search_session_history` Tool 语义检索。

## 可用 References
当需要更多详情时，使用 Tool 查询：

| Reference | 描述 | 建议 Query时机 |
|-----------|------|---------------|
| `preferences` | 完整的用户偏好设置（响应风格 + 话题偏好 + 反馈历史） | 用户表达反馈 / 回答需要个性化调整 |
| `behaviors` | 观察到的行为模式详情 + 建议 | Agent 需要调整响应策略 |

## 可用自定义 Skill
当触发条件命中时，使用 `load_skill` Tool 加载：

| Skill | 描述 | 触发条件 |
|-------|------|---------|
| `interview_tips` | 面试技巧 | 模拟面试开始时 |
| `rag_deep` | RAG 深度解析 | RAG 深度讨论 |

## 使用指南
1. 本文档始终加载，提供概要信息
2. 当概要不够详细时，调用 `load_memory_reference("{reference_name}")`
3. 当需要语义检索历史时，调用 `search_session_history("{query}")`
4. 当触发自定义 Skill 时，调用 `load_skill("{skill_name}")`
```

### 3.2 长度控制

**目标**：MEMORY.md 控制在 **300-500 tokens**

- 偏好概要：~50 tokens（3-5 条要点）
- 行为模式概要：~100 tokens（3-4 条要点）
- 会话历史概要：~100 tokens（从 session_summaries 动态查询最近 5 条）
- References 列表：~100 tokens（表格形式）
- Skill 列表：~50 tokens（表格形式）

### 4.2 behaviors.md

```markdown
# 用户行为模式详情

## 提问序列
用户经常在问"什么是 X"之后追问"为什么需要 X"或"X 怎么实现"。

**示例**：
- 用户问"什么是 Agent"时，回答后补充"Agent 的价值在于..."和"Agent 的实现方式..."

用户偏好 **practical_use**（实践应用）类型的答案，而非纯理论阐述。

- 检索时优先返回带代码示例的题目
- 回答时优先给出实践案例，原理部分简洁

用户的追问通常针对 **具体细节**（参数、边界情况、错误处理），而非宏观对比。

**证据**：用户追问"阈值怎么设置"、"召回失败怎么办"，而非"和其他方案对比"。

**建议**：回答时主动覆盖常见边界情况、错误处理方式。

## 知识背景推测
用户有一定 Python 基础，对 LangChain API 有使用经验，但对架构设计理解不深。

**建议**：回答时可以跳过 Python 基础解释，但架构层面要详细展开。

## 五、session_summaries 表设计

### 5.1 设计理念

**核心定位**：
> session_summaries 是 conversations 的增强层，不是独立的会话记录。

**职责划分**：

| 表 | 职责 | 存储内容 |
|---|------|---------|
| **conversations** | 对话管理 | title, created_at, updated_at |
| **messages** | 消息存储 | 完整的对话消息内容 |
| **session_summaries** | 检索 | summary, embedding |

**避免冗余**：
- 不存储 title（已有于 conversations）
- 不存储 timestamp（已有于 conversations）
- 不存储完整消息（已有于 messages）

### 5.2 表结构

```sql
CREATE TABLE session_summaries (
    conversation_id VARCHAR(36) PRIMARY KEY,
    user_id VARCHAR(36) NOT NULL,
    summary TEXT NOT NULL,
    embedding vector(1024),
    session_type VARCHAR(20) DEFAULT 'chat',
    metadata JSONB,
    created_at TIMESTAMP NOT NULL DEFAULT NOW(),
    FOREIGN KEY (conversation_id) REFERENCES conversations(id) ON DELETE CASCADE
);

-- 索引
CREATE INDEX idx_session_summaries_user_id ON session_summaries(user_id);
CREATE INDEX idx_session_summaries_user_created ON session_summaries(user_id, created_at DESC);

-- 向量索引（使用 pgvector）
CREATE INDEX idx_session_summaries_embedding ON session_summaries 
USING ivfflat (embedding vector_cosine_ops) WITH (lists = 100);
```

### 5.3 字段说明

| 字段 | 类型 | 说明 | 来源 |
|------|------|------|------|
| conversation_id | VARCHAR(36) | 主键，关联 conversations | 已有 |
| user_id | VARCHAR(36) | 用户 ID | 已有 |
| summary | TEXT | 会话摘要（包含关键问题、用户反馈） | LLM 生成 |
| embedding | vector(1536) | 语义向量（用于检索） | embedding model |
| session_type | VARCHAR(20) | 类型：'chat' / 'interview' | 自动判断 |
| metadata | JSONB | 元数据（面试得分、公司等） | 业务数据 |
| created_at | TIMESTAMP | 创建时间 | 系统生成 |

**设计说明**：
- **summary**：由 LLM 一次性生成，包含主题、关键问题、用户反馈，无需额外字段
- **embedding**：基于 summary 生成，用于语义检索，无需 keywords 硬过滤

### 5.5 CASCADE DELETE 优势

**用户预期**：
> 删除对话时，相关的记忆（摘要、向量）也应该被删除。

**实现**：
- 通过 `ON DELETE CASCADE` 外键约束
- 删除 conversation → 自动删除 session_summaries
- 无需手动清理，保证数据一致性

## 六、记忆检索与注入机制

### 6.1 整体流程

```
┌─────────────────────────────────────────────────────────────┐
│ 用户发起有效提问（≥10字，非空）                                │
│                                                              │
│ 判断：是否正在检索记忆？                                       │
│   - 是 → 本次不触发记忆检索                                    │
│   - 否 → 发起异步 Worker 检索记忆                              │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│ 异步 Worker：检索记忆                                         │
│                                                              │
│ 1. 语义检索 session_summaries（top_k = 5，可配置）            │
│ 2. 结果存入 Checkpointer 状态                                 │
│ 3. 标记：检索完成                                              │
│                                                              │
│ 注：仅检索 session_summaries，不自动加载 references           │
│    preferences/behaviors 采用渐进式披露，LLM 按需调用 Tool     │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│ 下次对话开始前                                                 │
│                                                              │
│ 检查：上次记忆是否检索完成？                                    │
│   - 完成 → 去重后注入 memory_context                           │
│   - 未完成 → 跳过注入，直接响应（不阻塞主流程）                  │
│                                                              │
│ 去重：使用 conversation_id                                    │
│ 容量控制：memory_context 上限 20kb，超过时裁剪最早记忆          │
│ 存储：状态持久化到 LangGraph Checkpointer                      │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│ 构建上下文                                                    │
│                                                              │
│ [System Message + MEMORY.md] ← MEMORY.md 始终加载             │
│ [对话历史]                                                    │
│ [memory_context（检索到的记忆）] ← 检索完成时注入               │
│ [当前用户问题]                                                │
│                                                              │
│ LLM 可按需调用：                                              │
│   - load_memory_reference("preferences")                     │
│   - load_memory_reference("behaviors")                       │
│   - search_session_history("query")                          │
│   - load_skill("skill_name")                                 │
└─────────────────────────────────────────────────────────────┘
```

### 6.2 触发条件

**有效提问判断**：
- 长度 ≥ 10 个字符
- 非空字符串（去除空白后）
- 每次有效提问都触发记忆检索

```python
def should_trigger_memory_retrieval(message: str) -> bool:
    """判断是否触发记忆检索"""
    
    # 基本条件：有效提问
    stripped = message.strip()
    if len(stripped) < 10:
        return False
    
    return True
```

### 6.2.1 配置项

**记忆检索配置**（可通过环境变量或配置文件调整）：

```python
# backend/app/config/settings.py

class Settings(BaseSettings):
    # 记忆检索配置
    memory_retrieval_min_length: int = 10       # 触发检索的最小消息长度
    memory_retrieval_top_k: int = 5             # 每次检索返回的最大记忆条数
    memory_retrieval_lock_timeout: int = 60     # 检索锁超时时间（秒）
    memory_context_max_size: int = 20 * 1024    # 记忆上下文最大窗口大小（20kb）
```

### 6.2.2 检索内容

每次有效提问触发检索时，**仅检索 session_summaries**（数据层）。

**preferences.md 和 behaviors.md** 采用渐进式披露原则：
- MEMORY.md 概要始终可见
- LLM 根据概要判断是否需要详情 → 主动调用 `load_memory_reference` Tool
- 不自动加载完整 references（节省 token 预算）

### 6.2.3 状态存储位置

记忆相关状态存储在 **LangGraph Checkpointer**（PostgreSQL）：

```python
class AgentState(TypedDict):
    messages: list[BaseMessage]
    memory_context: str                    # 已注入的记忆内容
    injected_session_ids: list[str]        # 已注入的会话 ID（去重用，上限 50）
```

**选择 Checkpointer 的原因**：
- 需要恢复现场（用户刷新页面后继续对话）
- 与 messages 状态统一管理
- conversation_id 作为 thread_id 自动持久化
- 无需手动清理，随 workflow state 自动管理

**Worker 与 Checkpointer 的交互**：
- Worker 检索完成后，直接调用 `checkpointer.aput()` 更新 memory_context
- 无需中间缓存，状态直接持久化
- 主 Agent 下次对话时，从 Checkpointer 读取最新 memory_context

### 6.2.4 检索未完成时的处理

当用户连续快速提问，第一次触发的检索尚未完成时：

```
用户: "RAG 的原理是什么？"     → 触发异步检索（耗时 2-3 秒）
用户: "召回阈值怎么设置？"     ← 检索尚未完成，memory_context 为空或旧值
```

**处理策略**：直接使用当前 memory_context（可能为空），不阻塞响应。

**设计优势**：
- Worker 更新是异步的，不影响主流程响应速度
- 检索完成后，memory_context 已更新，下次对话时自动使用新记忆
- 记忆是增强功能，不应阻塞用户响应

### 6.3 并发控制：Redis 锁

**锁粒度**：会话级别（conversation_id）

**意义**：
- 用户在设备 A 进行会话 abc → 可以检索记忆
- 同时用户在设备 B 进行会话 xyz → 也可以检索记忆（不同会话，独立）
- 同一个会话内，不重复触发检索

```python
from app.config.settings import get_settings


def get_memory_retrieval_lock_key(user_id: str, conversation_id: str) -> str:
    """获取记忆检索锁的 Key"""
    return f"memory_retrieval:{user_id}:{conversation_id}"


async def acquire_memory_retrieval_lock(user_id: str, conversation_id: str) -> bool:
    """获取记忆检索锁
    
    Returns:
        True - 成功获取锁，可以开始检索
        False - 锁已存在，正在检索中，本次不触发
    """
    settings = get_settings()
    redis_client = get_redis_client()
    lock_key = get_memory_retrieval_lock_key(user_id, conversation_id)
    
    # 使用 SETNX 实现分布式锁
    acquired = redis_client.setnx(lock_key, "locked")
    if acquired:
        redis_client.expire(lock_key, settings.memory_retrieval_lock_timeout)
        logger.debug(f"Memory retrieval lock acquired: {lock_key}")
        return True
    
    logger.debug(f"Memory retrieval lock exists: {lock_key}, skip this round")
    return False


async def release_memory_retrieval_lock(user_id: str, conversation_id: str) -> None:
    """释放记忆检索锁"""
    redis_client = get_redis_client()
    lock_key = get_memory_retrieval_lock_key(user_id, conversation_id)
    redis_client.delete(lock_key)
    logger.debug(f"Memory retrieval lock released: {lock_key}")
```

### 6.4 容量控制与去重

**原理**：
- 使用 `conversation_id` 作为唯一标识进行去重
- 使用**内存大小**（而非数量）控制记忆窗口上限
- 超过上限时，裁剪最早的记忆内容

**实现位置**：容量控制逻辑已内置在 Worker 的 `merge_memory_context()` 函数中（见 6.5 节）。

**配置项**：
- `memory_context_max_size`：默认 20kb，超过时裁剪最早记忆

### 6.5 异步检索 Worker

```python
from app.config.settings import get_settings
from langgraph.checkpoint.postgres import PostgresSaver

async def retrieve_memory_worker(
    user_id: str,
    conversation_id: str,
    query: str,
    checkpointer: PostgresSaver,
):
    """异步检索记忆
    
    Args:
        user_id: 用户 ID
        conversation_id: 会话 ID（作为 thread_id）
        query: 用户问题（用于语义检索）
        checkpointer: LangGraph Checkpointer 实例
    """
    settings = get_settings()
    
    try:
        # 1. 计算 query embedding
        query_embedding = get_embedding_tool().embed_text(query)
        
        # 2. 语义检索 session_summaries
        results = pg.search_session_summaries(
            user_id=user_id,
            query_embedding=query_embedding,
            top_k=settings.memory_retrieval_top_k,  # 默认 5
        )
        
        # 3. 获取对话标题（从 conversations 表）
        for r in results:
            conv = pg.get_conversation(user_id, r["conversation_id"])
            if conv:
                r["title"] = conv.title
                r["created_at"] = conv.created_at
        
        # 4. 格式化检索结果为 memory_context
        new_context = format_memory_context(results)
        
        # 5. 读取当前 checkpoint，更新 memory_context
        config = {"configurable": {"thread_id": conversation_id}}
        current_checkpoint = await checkpointer.aget(config)
        
        if current_checkpoint:
            # 获取当前状态
            current_memory = current_checkpoint.channel_values.get("memory_context", "")
            injected_ids = current_checkpoint.channel_values.get("injected_session_ids", [])
            
            # 去重并合并
            merged_context, merged_ids = merge_memory_context(
                current_memory, injected_ids, results
            )
            
            # 更新 checkpoint
            current_checkpoint.channel_values["memory_context"] = merged_context
            current_checkpoint.channel_values["injected_session_ids"] = merged_ids
            
            await checkpointer.aput(config, current_checkpoint)
        
        logger.info(f"Memory retrieval completed for {conversation_id}, found {len(results)} sessions")
        
    except Exception as e:
        logger.error(f"Memory retrieval failed: {e}")
    
    finally:
        # 释放锁
        await release_memory_retrieval_lock(user_id, conversation_id)


def format_memory_context(results: list[dict]) -> str:
    """格式化检索结果为 memory_context
    
    Args:
        results: 检索到的 session_summaries 列表
    
    Returns:
        格式化后的记忆上下文（Markdown 格式）
    """
    sessions = []
    for r in results:
        title = r.get("title", "")
        summary = r.get("summary", "")
        created_at = r.get("created_at", "")
        
        sessions.append(f"### {title} ({created_at.strftime('%Y-%m-%d')})\n{summary}")
    
    return "\n\n".join(sessions)


def merge_memory_context(
    current_memory: str,
    injected_ids: list[str],
    new_results: list[dict],
) -> tuple[str, list[str]]:
    """合并新旧记忆上下文（带去重和容量控制）
    
    Args:
        current_memory: 当前 memory_context
        injected_ids: 当前已注入的会话 ID
        new_results: 新检索到的 session_summaries
    
    Returns:
        (merged_memory, merged_ids)
    """
    settings = get_settings()
    max_size = settings.memory_context_max_size  # 默认 20kb
    
    # 去重
    new_sessions = []
    for r in new_results:
        conversation_id = r["conversation_id"]
        
        if conversation_id in injected_ids:
            continue
        
        session_text = f"### {r.get('title', '')}\n{r.get('summary', '')}"
        new_sessions.append(session_text)
        injected_ids.append(conversation_id)
    
    # 合并
    if new_sessions:
        new_context = "\n\n".join(new_sessions)
        if current_memory:
            merged_memory = current_memory + "\n\n" + new_context
        else:
            merged_memory = new_context
    else:
        merged_memory = current_memory
    
    # 容量控制：超过 max_size 时裁剪最早的记忆
    current_size = len(merged_memory.encode('utf-8'))
    
    if current_size > max_size:
        sessions = merged_memory.split("\n\n### ")
        
        while len(merged_memory.encode('utf-8')) > max_size and len(sessions) > 1:
            removed_session = sessions.pop(0)
            if injected_ids:
                injected_ids.pop(0)
            logger.info(f"Memory context exceeded {max_size}b, removed earliest session")
        
        if sessions:
            merged_memory = "\n\n### ".join(sessions)
        else:
            merged_memory = ""
    
    return merged_memory, injected_ids

## 九、并发控制

### 9.1 问题分析

**并发场景**：
- 多个对话同时结束，多个 Worker 同时更新同一用户的 preferences.md、behaviors.md
- LLM 编辑文档需要时间，后一个 Worker 可能覆盖前一个 Worker 的修改

**风险**：
- 丢失部分用户反馈
- behaviors.md 状态不稳定

### 9.2 解决方案：Redis 分布式锁

```python
import redis
from app.db.redis_client import get_redis_client

MEMORY_LOCK_TIMEOUT = 30  # 锁超时时间（秒）


async def acquire_memory_lock(user_id: str, lock_type: str) -> bool:
    """获取记忆更新锁
    
    Args:
        user_id: 用户 ID
        lock_type: 锁类型（"preferences" / "behaviors" / "memory_index"）
    
    Returns:
        是否成功获取锁
    """
    redis_client = get_redis_client()
    lock_key = f"memory_lock:{user_id}:{lock_type}"
    
    # 使用 SETNX 实现分布式锁
    acquired = redis_client.setnx(lock_key, "locked")
    if acquired:
        redis_client.expire(lock_key, MEMORY_LOCK_TIMEOUT)
        logger.debug(f"Lock acquired: {lock_key}")
        return True
    
    logger.debug(f"Lock not acquired: {lock_key}, retry later")
    return False


async def release_memory_lock(user_id: str, lock_type: str) -> None:
    """释放记忆更新锁"""
    redis_client = get_redis_client()
    lock_key = f"memory_lock:{user_id}:{lock_type}"
    redis_client.delete(lock_key)
    logger.debug(f"Lock released: {lock_key}")


async def update_with_lock(
    user_id: str,
    lock_type: str,
    update_func: Callable,
    *args,
    **kwargs,
) -> None:
    """带锁的更新操作
    
    Args:
        user_id: 用户 ID
        lock_type: 锁类型
        update_func: 更新函数
    """
    max_retries = 3
    retry_delay = 5  # 重试间隔（秒）
    
    for attempt in range(max_retries):
        if await acquire_memory_lock(user_id, lock_type):
            try:
                await update_func(*args, **kwargs)
            finally:
                await release_memory_lock(user_id, lock_type)
            return
        
        # 未获取锁，等待后重试
        if attempt < max_retries - 1:
            logger.info(f"Lock retry {attempt + 1}/{max_retries} for {lock_type}")
            await asyncio.sleep(retry_delay)
    
    # 重试失败，放弃本次更新
    logger.warning(f"Lock acquisition failed after {max_retries} retries, skip update")
```

    """从对话分析用户行为（带锁）"""
    
    async def _update():
        messages = pg.get_messages(user_id, conversation_id)
        current = read_memory_reference(user_id, "behaviors")
        
        prompt = f"""
当前用户行为模式：
{current}

本次对话：
{format_messages(messages)}

请分析并更新行为模式文档...
"""
        
        updated = await llm.ainvoke(prompt)
        write_memory_reference(user_id, "behaviors", updated)
    
    await update_with_lock(user_id, "behaviors", _update)

## 十、错误处理与降级

### 10.1 重试机制

**重试策略**：
- 最大重试次数：3 次
- 重试间隔：递增（1s → 3s → 5s）
- 重试触发条件：LLM 调用失败、embedding 计算失败、数据库写入失败

```python
MAX_RETRIES = 3
RETRY_DELAYS = [1, 3, 5]  # 秒


async def retry_with_backoff(
    func: Callable,
    *args,
    **kwargs,
) -> tuple[bool, Any]:
    """带退避的重试
    
    Returns:
        (success, result_or_error)
    """
    last_error = None
    
    for attempt in range(MAX_RETRIES):
        try:
            result = await func(*args, **kwargs)
            return (True, result)
        except Exception as e:
            last_error = e
            logger.warning(f"Attempt {attempt + 1} failed: {e}")
            
            if attempt < MAX_RETRIES - 1:
                delay = RETRY_DELAYS[attempt]
                logger.info(f"Retry in {delay} seconds...")
                await asyncio.sleep(delay)
    
    return (False, last_error)
```

### 10.2 降级策略

**降级场景**：summary 生成失败后，使用 conversation 表的 title 作为摘要

```python
async def save_session_summary(conversation_id: str, user_id: str):
    """对话结束时，写入 session_summaries（带重试和降级）"""
    
    messages = pg.get_messages(user_id, conversation_id)
    conv = pg.get_conversation(user_id, conversation_id)
    
    # 1. 尝试生成 summary（重试 3 次）
    success, summary = await retry_with_backoff(generate_summary, messages)
    
    if not success:
        # 降级：使用 conversation title 作为 summary
        logger.warning(f"Summary generation failed, fallback to conversation title")
        summary = f"主题：{conv.title}"
    
    # 2. 尝试计算 embedding（重试 3 次）
    success, embedding = await retry_with_backoff(
        get_embedding_tool().embed_text,
        summary
    )
    
    if not success:
        # 降级：不写入 embedding，允许 NULL
        logger.warning(f"Embedding generation failed, write without embedding")
        embedding = None
    
    # 3. 写入 session_summaries（重试 3 次）
    session_type = detect_session_type(messages)
    metadata = {}
    if session_type == "interview":
        metadata = get_interview_metadata(conversation_id) or {}
    
    success, error = await retry_with_backoff(
        pg.create_session_summary,
        conversation_id,
        user_id,
        summary,
        embedding,
        session_type,
        metadata,
    )
    
    if not success:
        logger.error(f"Failed to write session_summary after retries: {error}")
        # 放弃本次写入，不影响用户使用
        return
    
    logger.info(f"Session summary saved: {conversation_id}")
    
    # 4. 尝试同步反馈到 preferences.md（失败不影响主流程）
    if summary and "用户反馈" in summary:
        feedback = extract_feedback_from_summary(summary)
        if feedback:
            await update_preferences_from_feedback(user_id, feedback)
```

### 10.3 错误分类

| 错误类型 | 处理方式 | 影响范围 |
|---------|---------|---------|
| **LLM 调用失败** | 重试 3 次 → 降级使用 title | 仅影响 summary 质量 |
| **Embedding 失败** | 重试 3 次 → 写入 NULL | 影响语义检索，不影响其他功能 |
| **数据库写入失败** | 重试 3 次 → 放弃本次写入 | 丢失本次会话摘要，不影响对话 |
| **锁获取失败** | 重试 3 次 → 放弃本次更新 | preferences/behaviors 暂不更新 |

## 十一、存储实现

### 11.1 PostgresStore（references 文件）

所有 references 文件存储在 PostgresStore：

```
Namespace 结构：

("memory", user_id) → MEMORY.md
    {"content": memory_md_text}

("memory", user_id, "references", "preferences") → preferences.md
    {"content": preferences_md_text}

("memory", user_id, "references", "behaviors") → behaviors.md
    {"content": behaviors_md_text}

("memory", user_id, "references", "skills", skill_name, "SKILL.md") → SKILL.md
    {"content": skill_md_text}

("memory", user_id, "references", "skills", skill_name, "references", ref_name) → reference 文件
    {"content": ref_md_text}
```

### 11.2 数据库表（session_summaries）

session_summaries 使用独立的 PostgreSQL 表（而非 PostgresStore），便于：
- 使用 pgvector 进行向量检索
- CASCADE DELETE 保证数据一致性
- 高效的 SQL 查询

### 11.3 读写接口

```python
def read_memory(user_id: str) -> str:
    """读取 MEMORY.md"""
    result = store.get(("memory", user_id), "MEMORY.md")
    return result.value["content"] if result else ""


def write_memory(user_id: str, content: str) -> None:
    """写入 MEMORY.md"""
    store.put(("memory", user_id), "MEMORY.md", {"content": content})


def read_memory_reference(user_id: str, reference_name: str) -> str:
    """读取 references 文件"""
    result = store.get(("memory", user_id, "references"), reference_name)
    return result.value["content"] if result else ""


def write_memory_reference(user_id: str, reference_name: str, content: str) -> None:
    """写入 references 文件"""
    store.put(("memory", user_id, "references"), reference_name, {"content": content})


# session_summaries 使用 PostgreSQL 表
class PostgresClient:
    
    def create_session_summary(
        self,
        conversation_id: str,
        user_id: str,
        summary: str,
        embedding: list[float],
        session_type: str,
        metadata: dict,
    ) -> None:
        """写入 session_summaries"""
        with self.conn.cursor() as cur:
            cur.execute(
                """
                INSERT INTO session_summaries
                (conversation_id, user_id, summary, embedding, session_type, metadata)
                VALUES (%s, %s, %s, %s, %s, %s)
                """,
                (conversation_id, user_id, summary, embedding, 
                 session_type, json.dumps(metadata)),
            )
            self.conn.commit()
    
    def search_session_summaries(
        self,
        user_id: str,
        query_embedding: list[float],
        top_k: int = 3,
    ) -> list[dict]:
        """语义检索 session_summaries"""
        with self.conn.cursor(cursor_factory=RealDictCursor) as cur:
            cur.execute(
                """
                SELECT conversation_id, summary,
                       1 - (embedding <=> %s::vector) as similarity
                FROM session_summaries
                WHERE user_id = %s
                ORDER BY embedding <=> %s::vector
                LIMIT %s
                """,
                (query_embedding, user_id, query_embedding, top_k),
            )
            return cur.fetchall()
    
    def get_recent_session_summaries(
        self,
        user_id: str,
        limit: int = 5,
    ) -> list[dict]:
        """获取最近 N 条 session_summaries（用于 MEMORY.md 概要）"""
        with self.conn.cursor(cursor_factory=RealDictCursor) as cur:
            cur.execute(
                """
                SELECT ss.conversation_id, ss.summary, ss.session_type, ss.metadata,
                       c.title, c.created_at
                FROM session_summaries ss
                JOIN conversations c ON ss.conversation_id = c.id
                WHERE ss.user_id = %s
                ORDER BY c.created_at DESC
                LIMIT %s
                """,
                (user_id, limit),
            )
            return cur.fetchall()
```

## 十七、总结

### 17.1 核心设计决策

| 决策 | 原因 |
|------|------|
| **记忆 = Skill** | 遵循行业标准，概念统一 |
| **MEMORY.md + references** | 渐进式披露，节省 token |
| **session_summaries 独立表** | 数据 vs 规则分离，支持检索 |
| **CASCADE DELETE** | 删除对话时自动清理记忆，符合用户预期 |
| **Skills 必须遵循标准** | 兼容 OpenAI/Claude Skills 生态 |
| **状态存储到 Checkpointer** | 恢复现场，与 messages 统一管理 |
| **检索配置项化** | top_k = 5，便于后续调优 |
| **preferences/behaviors 渐进式披露** | LLM 按需调用 Tool，不自动加载 |
| **检索未完成时跳过注入** | 不阻塞主流程，记忆是增强功能 |
| **记忆窗口大小控制** | 20kb 上限，超过时裁剪最早记忆 |

### 17.2 数据边界清晰

| 类型 | 存储方式 | 用途 |
|------|---------|------|
| **规则**（preferences/behaviors） | Markdown 文件 | 指导 Agent 行为 |
| **数据**（session_summaries） | 数据库表 | 语义检索 |
| **消息**（messages） | 数据库表 | 完整历史 |
| **运行时状态**（memory_context 等） | Checkpointer | 恢复现场 |

### 17.3 检索配置一览

| 配置项 | 默认值 | 说明 |
|--------|-------|------|
| `memory_retrieval_min_length` | 10 | 触发检索的最小消息长度 |
| `memory_retrieval_top_k` | 5 | 每次检索返回的最大记忆条数 |
| `memory_retrieval_lock_timeout` | 60 | 检索锁超时时间（秒） |
| `memory_context_max_size` | 20kb | 记忆上下文最大窗口大小 |
