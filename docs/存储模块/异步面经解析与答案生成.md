# 异步面经解析与答案生成

## 一、整体架构

```
┌─────────────────────────────────────────────────────────────────────┐
│                        用户上传面经                                  │
└─────────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────────┐
│  POST /extract/submit                                                │
│  → ExtractTaskService.submit()                                       │
│  → PostgreSQL: ExtractTask (status=pending)                          │
└─────────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────────┐
│  Extract Worker（轮询 PostgreSQL）                                   │
│  → 获取 status=pending 的任务                                        │
│  → Vision Extractor 解析                                             │
│  → PostgreSQL: ExtractTask (status=completed, result=...)            │
└─────────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────────┐
│  POST /extract/tasks/{id}/confirm                                    │
│  → ExtractTaskService.confirm()                                      │
│  → IngestionService.ingest_interview()                               │
└─────────────────────────────────────────────────────────────────────┘
                              ↓
        ┌─────────────────────┴─────────────────────┐
        ↓                                           ↓
┌───────────────────────┐                 ┌───────────────────────┐
│  Qdrant               │                 │  RabbitMQ             │
│  Question 入库        │                 │  MQTaskMessage        │
│  (answer=None)        │                 │  (触发答案生成)        │
└───────────────────────┘                 └───────────────────────┘
        ↓                                           ↓
        │                                  ┌───────────────────────┐
        │                                  │  Answer Worker        │
        │                                  │  → LLM 生成答案        │
        │                                  │  → Qdrant 更新 answer │
        │                                  └───────────────────────┐
        ↓                                           ↓
┌─────────────────────────────────────────────────────────────────────┐
│  Qdrant: Question (answer="生成的答案")                              │
└─────────────────────────────────────────────────────────────────────┘
```

## 二、存储分布

| 数据 | 存储位置 | 用途 |
|------|---------|------|
| **ExtractTask** | PostgreSQL | 任务状态追踪（pending→processing→completed→confirmed） |
| **Question** | Qdrant | 面经题目（含 answer 字段） |
| **MQTaskMessage** | RabbitMQ | 答案生成任务触发 |

## 三、两个异步通道

| 通道 | 触发方式 | 处理内容 | 实现文件 |
|------|---------|---------|---------|
| **Extract Worker** | 轮询 PostgreSQL | Vision Extractor 解析面经 | `extract_worker.py` |
| **Answer Worker** | 消费 RabbitMQ | LLM 生成答案 | `answer_worker.py` |

### 设计差异分析

| Worker | 模式 | 原因 |
|--------|------|------|
| **Extract Worker** | 轮询 PostgreSQL | 任务数据已在 PostgreSQL，轮询简单可靠，无需额外发 MQ |
| **Answer Worker** | 消费 RabbitMQ | 需要高并发处理（LLM 调用阻塞型），MQ 提供吞吐量控制 |

**Extract Worker 不用 MQ 的原因**：
- ExtractTask 提交时已经写入 PostgreSQL
- PostgreSQL 是可靠存储，轮询模式不会丢失任务
- 避免引入额外的 MQ 依赖

**Answer Worker 用 MQ 的原因**：
- 需要从主系统触发（入库后发消息）
- MQ 提供更好的吞吐量和并发控制
- Answer Worker 用线程池处理阻塞型 LLM 调用

## 四、状态流转

### ExtractTask 状态

```
pending → processing → completed → confirmed
                  ↓
                failed（可重试）
```

| 状态 | 含义 | 后续操作 |
|------|------|---------|
| `pending` | 待处理 | Worker 获取任务 |
| `processing` | 正在解析 | Vision Extractor 执行 |
| `completed` | 解析完成 | 用户确认入库 |
| `confirmed` | 已入库 | 流程结束 |
| `failed` | 解析失败 | 可手动重试 |

### Question 状态（隐式）

```
answer = None → answer = "生成的答案"
```

入库时 `answer=None`，Answer Worker 消费 MQ 后填充。

### MQTaskMessage 状态（隐式）

```
sent → consumed → processed
              ↓
            retry → DLQ（超过重试阈值）
```

## 五、一致性风险点

### 问题：Qdrant + RabbitMQ 无法同一事务

```
风险点：入库 → MQ 发送
┌─────────────────────────────────────────┐
│  IngestionService.ingest_interview()    │
├─────────────────────┬───────────────────┤
│  Qdrant: Question   │  RabbitMQ: 发送   │
│  answer=None        │  可能失败！       │
└─────────────────────┴───────────────────┘
         ↓                    ↓
      成功写入              MQ 丢失
         ↓                    ↓
    Question 永久无答案
```

**根本原因**：
- Question 存储在 Qdrant（向量数据库）
- MQTaskMessage 发送到 RabbitMQ
- 无法用单一事务同时覆盖两者

**传统 Outbox Pattern 的前提**：
- 业务数据 + 消息记录 → 同一个数据库
- 利用本地事务的原子性保证一致性
- 本项目：Question 在 Qdrant，Outbox 只能在 PostgreSQL，无法共享事务

## 六、补偿机制

### 补偿脚本设计

**文件**：`scripts/resend_to_queue.py`

**扫描条件**：
- Qdrant: `Question.answer = None`
- `QuestionType ∈ {KNOWLEDGE, SCENARIO, ALGORITHM}`（需要异步答案的类型）

**操作流程**：
- 重新构建 MQTaskMessage
- 发送到 RabbitMQ
- Answer Worker 重新消费

**运行方式**：
```bash
# 预览模式（只显示不发送）
PYTHONPATH=. uv run python scripts/resend_to_queue.py --dry-run

# 实际发送
PYTHONPATH=. uv run python scripts/resend_to_queue.py
```

### 幂等性保证

Answer Worker 的幂等设计（`answer_worker.py`）：

```python
def process_answer_task(task):
    # 幂等检查：先判断答案是否已存在
    existing = question_repo.find_by_id(task.question_id)
    if existing and existing.answer:
        logger.info("Answer already exists, skipping")
        return True  # 跳过，但返回成功（消息消费成功）

    # 生成答案
    answer = agent.generate_answer(question)
    question_repo.update_answer(question_id, answer)
```

**效果**：补偿脚本重发消息时，已处理的题目不会重复生成答案。

## 八、潜在优化方向

| 方案 | 改动 | 优点 | 缺点 |
|------|------|------|------|
| **A. Outbox + 先写 PostgreSQL** | Question 也存 PostgreSQL，Outbox 同事务 | 强一致性保证 | 需重构入库流程，Qdrant 变为异步同步 |
| **B. 自动定时补偿** | 定时任务自动执行 resend | 无需手动干预 | 可能重复处理，需监控告警 |
| **C. 入库状态追踪表** | PostgreSQL 添加 IngestionRecord | 可追踪入库状态 | 需新增表，仍需补偿 |

**当前选择**：手动补偿脚本（方案 B 的简化版）

## 十、架构反思：Qdrant vs pgvector

### 问题背景

当前架构使用 Qdrant + PostgreSQL 双存储，Question 存储在 Qdrant，ExtractTask 存储在 PostgreSQL。这导致无法用同一事务保证一致性，需要补偿脚本来兜底。

**面试常见问题**：为什么选择 Qdrant 而不是 pgvector？

### 对比分析

| 维度 | pgvector | Qdrant |
|------|----------|--------|
| **一致性** | 可与 PostgreSQL 同事务 | 跨数据库，无法同事务 |
| **运维复杂度** | 单一数据库 | 双数据库依赖 |
| **数据量适用** | 十万级以内 | 百万级以上 |
| **索引类型** | HNSW, IVFFlat | HNSW, 多种量化 |
| **过滤能力** | SQL WHERE | Payload 过滤（类似） |
| **本项目需求** | ✅ 足够 | 过度 |

### 性能数据参考

- pgvector HNSW 在 10万向量时，检索延迟约 10-50ms
- 本项目预估题目数量：万级（远低于瓶颈）
- 性能差距在百万级数据才明显

### 架构决策反思

**当时的思维误区**：

```
"向量数据库"听起来更专业 → 选择 Qdrant
        ↓
忽略了实际数据量规模
        ↓
引入双存储复杂度
        ↓
无法同一事务 → 一致性风险 → 需要补偿脚本
```

**正确的决策流程应该是**：

```
1. 数据量预估 → 万级
2. 性能需求 → pgvector 10万级足够
3. 一致性需求 → 单数据库更简单
4. 运维成本 → 单一数据库更低
   → 结论：pgvector
```

### 如果使用 pgvector 重构

**表设计**：

```sql
-- PostgreSQL 表设计（Question 也存 PostgreSQL）
CREATE TABLE questions (
    question_id VARCHAR(32) PRIMARY KEY,
    question_text TEXT,
    company VARCHAR(100),
    position VARCHAR(100),
    question_type VARCHAR(20),
    answer TEXT,
    embedding VECTOR(1536),  -- pgvector
    created_at TIMESTAMP
);

-- HNSW 索引
CREATE INDEX idx_questions_embedding 
ON questions USING hnsw (embedding vector_cosine_ops);

-- Outbox 表（同事务）
CREATE TABLE outbox_messages (
    id UUID PRIMARY KEY,
    aggregate_type VARCHAR(50),
    aggregate_id VARCHAR(32),
    payload JSONB,
    status VARCHAR(20),  -- pending/sent
    created_at TIMESTAMP
);
```

**入库流程（单一事务）**：

```python
async def ingest_question(question):
    async with db.transaction():
        # 1. Question + Outbox 同事务
        db.insert("questions", question)
        db.insert("outbox_messages", {
            "aggregate_id": question.question_id,
            "payload": mq_message,
            "status": "pending"
        })
    
    # 2. Poller 异步发送 MQ（失败可重试）
```

**效果**：
- Question 和 Outbox 在同一 PostgreSQL 事务
- Outbox Pattern 可直接应用
- MQ 发送失败时，消息记录保留，可重新发送
- 不需要跨数据库补偿脚本

### 面试回答建议

**诚实回答**：

> 目前项目使用 Qdrant + PostgreSQL 双存储，确实存在过度设计的嫌疑。
>
> pgvector 作为 PostgreSQL 插件，可以让 Question 和 ExtractTask 在同一个数据库，避免跨数据库事务问题。当前的一致性风险（入库后 MQ 丢失）就是双存储带来的。
>
> 选择 Qdrant 的初衷是认为"向量检索性能更好"，但实际上项目数据量在万级，pgvector 完全够用。双存储带来的架构复杂度远大于性能收益。

**展示思考深度**：

> 这是个架构权衡问题，我当时的决策有改进空间。
>
> **pgvector 方案的优势**：
> - Question 和 ExtractTask 同一数据库 → 可用 Outbox Pattern 保证一致性
> - 运维简单，单一数据库依赖
> - PostgreSQL 16+ 的 pgvector 已支持 HNSW 索引，性能足够
>
> **如果重来，我会选择 pgvector**：
> - 题目入库 → PostgreSQL + Outbox → MQ 发送 → 强一致性保证
> - 向量检索 → pgvector HNSW 索引 → 性能足够
> - 单一数据库，运维简单

### 结论

**诚实承认**：当前架构确实有过度设计嫌疑，Qdrant 在项目实际数据规模下并非必要。
